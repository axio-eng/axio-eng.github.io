Epistemic Bridge — Grafo Multi-Relacional com Consciência Epistemológica ​

Tipo: Conceito / Research Note
Data: 25 Jun. 2026
Status: Pensamento — prototipação futura

1. Motivação ​

O NEXUS hoje opera com níveis de abstração (L0–L3) em um único eixo. Código é ingerido, abstraído semanticamente, e navegado topologicamente. Mas esse modelo unidimensional não captura a natureza epistemológica dos dados — quão certos estamos sobre cada informação, e como atravessar diferentes regimes de certeza dentro do mesmo grafo.

Este documento propõe um modelo onde cada nó e cada aresta carrega estado epistêmico, e a navegação entre domínios heterogêneos (código legado, schema SQL arbitrário, hipóteses de LLM) é mediada por pontes epistemológicas.

2. O Problema ​

2.1 Cenário Concreto ​

Nó A: schema SQL arbitrário, criado por LLM (epistemic = speculative)
Nó B: schema legado imutável, mal otimizado, sem RLS (epistemic = fact, cohesion = low)

Como caminhar de A para B com rigor? A aresta direta A→B tem baixa confiança (semantic_bridge com certainty ≈ 0.2). Mas um caminho via mediador:

A → (part_of) → Domain_A (fact) → (depends_on) → Domain_B (fact) → (part_of) → B (fact)

Tem certainty = 1.0 em cada aresta, mesmo que o caminho seja mais longo. O rigor do caminho depende do pior elo, não do mais curto.

2.2 Insuficiência do Modelo Atual ​

O T4 hoje tem 4 tipos de aresta (part_of, depends_on:imports, semantic_bridge, contradicts) e 3 eixos de projeção (criteria × file_type × project). Não há:

  • Metadado epistêmico no nó (quão certo estamos deste dado?)
  • Metadado de coesão (quão bem estruturado é este domínio?)
  • Pontes entre domínios com semântica de confiança
  • Caminhada parametrizada por min_certainty

3. O Modelo ​

3.1 Episteme como Eixo de Projeção ​

Cada nó ganha dois novos campos:

Node
├── domain: code | schema | document | hypothesis | bridge
├── epistemic: fact | inferred | hypothesized | speculative
└── cohesion: high (constraints, types, AST) | medium (ORM, semi-estruturado) | low (bruto, sem schema)

O eixo epistemic forma uma escala:

fact ───→ inferred ───→ hypothesized ───→ speculative
  ↑            ↑              ↑                ↑
  observado    deduzido       proposto         imaginado
  (parser,     (LLM sobre     (LLM sobre       (LLM livre,
   SQL DDL)     evidência)     padrão)           brainstorming)

3.2 Tensor de Distância ​

A distância entre dois nós não é um escalar — é um tensor quadridimensional:

dist(A, B) = (
    hops,              # contagem de arestas no caminho
    min_certainty,     # pior certainty entre todas as arestas
    epistemic_delta,   # custo de atravessar regimes de certeza:
                       #   fact→fact = 0.0
                       #   fact→speculative = 0.7
                       #   speculative→fact = 0.9 (sobe o rigor)
    cohesion_delta     # custo de atravessar coesão:
                       #   high→high = 0.0
                       #   high→low = 0.4
)

O consumidor (LLM, query) especifica restrições:

/query {
  "text": "como este schema se relaciona com o código legado?",
  "min_certainty": 0.7,
  "max_hops": 5
}

O runtime filtra caminhos que não atendem, e do que sobra, escolhe o mais curto ou o de maior min_certainty (parametrizável).

3.3 Bridge Schema ​

Um nó do tipo domain: bridge serve como mediador epistemológico entre dois domínios que não podem (ou não devem) se conectar diretamente:

Bridge_A_B
├── domain: bridge
├── epistemic: hypothesized (criado por LLM ou admin)
├── schema: mapping
│   ├── coluna_A → coluna_B
│   ├── conceito_X → conceito_Y
│   └── certainty: 0.8 (validado manualmente)
└── edges:
    → Nó_A via semantic_bridge, certainty=0.8
    → Nó_B via semantic_bridge, certainty=0.8

Propriedades:

  • A bridge é verificável: um agente pode auditá-la e aumentar/diminuir certainty
  • A bridge é mutável: se o schema do domínio A muda, a bridge pode ser revalidada
  • A bridge é decaível (ADR-0021): se ninguém atravessa, decay_score aumenta, e a bridge pode ser removida

Vantagem sobre aresta direta A→B:

Aresta diretaVia Bridge
AcoplamentoA e B precisam se conhecerA e B não sabem um do outro
ManutençãoMudança em A quebra A→BBridge absorve a mudança
AuditoriaSem ponto de verificaçãoBridge é auditável
ReusoSó A→B serveBridge pode conectar N domínios

3.4 Exemplo de Caminhada ​

LLM: "como este schema arbitrário se relaciona com o sistema legado?"

1. Query busca seed em T3 (Qdrant) → encontra Nó_A (speculative, cohesion=low)
2. T4 expande: Nó_A → (part_of) → Domain_A (fact, cohesion=high)
3. T4 expande: Domain_A → (depends_on) → Domain_B (fact, cohesion=low)
   → caminho directo, min_certainty=1.0, epistemic_delta=0.0 (fact→fact)
4. T4 expande: Domain_B → (part_of) → Nó_B (fact, cohesion=low)
5. Resultado: caminho com 3 hops, min_certainty=1.0

Alternativa:
1. T4 expande: Nó_A → (semantic_bridge) → Bridge_A_B → Nó_B
   → 2 hops, min_certainty=0.8

O LLM escolhe: caminho mais rigoroso (fact→fact, certainty=1.0, 3 hops)
ou caminho mais curto (via bridge, certainty=0.8, 2 hops).

5. Fractal e Auto-Similaridade ​

A intuição original: o padrão epistêmico se repete em diferentes escalas de escopo.

Escopo linha (L0)         Escopo função (L1)        Escopo módulo (L3)
fact ──→ inferred          fact ──→ inferred          fact ──→ inferred
│                            │                            │
▼                            ▼                            ▼
syntax error → provável     docstring ausente →          módulo sem ADR →
bug (inferred)              propósito inferido            propósito especulado

O mesmo padrão fact → inferred → hypothesized aparece em cada nível, apenas com granularidade diferente. Isso é auto-similaridade — uma propriedade fractal.

Mas implementar fractais literais (dimensão de Hausdorff, iterações de função) seria overengineering neste estágio. O tensor quadridimensional já captura a essência:

  • hops: a escala (quantos níveis de escopo atravessar?)
  • epistemic_delta: o custo de mudar de regime de certeza (a "densidade" da transição)
  • cohesion_delta: a qualidade estrutural do domínio (a "rugosidade" do espaço)

O que o tensor NÃO captura (e fractais fariam) é a predição de comportamento em escala não observada. Se conhecemos o padrão epistêmico em L0–L2, poderíamos inferir o comportamento em L5 sem precisar ingerir dados daquele nível. Isso é um problema de pesquisa aberto para o NEXUS.

6. O que NÃO é (para não confundir) ​

  • Não é um novo ADR. Nenhuma decisão foi tomada. É uma exploração.
  • Não substitui T3/T4. É uma camada conceitual sobre T4.
  • Não exige mudança no Qdrant. Qdrant continua servindo seeds por proximidade. A navegação epistemológica é no T4.
  • Não resolve migrations SQL tradicionais. Bridge é uma alternativa semântica, não substituta de DDL.

7. Próximos Passos (Quando Pertinente) ​

  1. Adicionar campo epistemic à tabela abstractions (text check: 'fact','inferred','hypothesized','speculative')
  2. Adicionar campo certainty à tabela agent_links (float 0.0–1.0)
  3. Prototipar NomadBridge: observa dois domínios, propõe semantic_bridge com certainty
  4. Estender /query com parâmetros min_certainty e max_hops
  5. Documentar casos de uso com bridges reais (schema legado → schema LLM)

"A distância entre dois pontos não é o que os separa, mas o que é preciso atravessar para entendê-los."
— Nota de arquitetura, 25 Jun. 2026
