Relatório de Integração: NEXUS → IntraClinica NG ​

Autor: Arquiteto de Sistemas (Agente Delegado)
Data: 22 de Maio de 2026
Classificação: Operacional — Leitura Obrigatória para Orquestradores e Workers

Diagnóstico do Problema ​

O fluxo Hive atual sofre de saturação de contexto por injeção bruta. O AGENTS.md com 900+ linhas é sintoma, não causa. A causa raiz é a ausência de um filtro de relevância entre o corpus de conhecimento e a janela de contexto do agente.

O NEXUS já tem todos os componentes para resolver isso:

  • nexus_context.py como motor de busca em linguagem natural (T3 → T4 → T2)
  • axio-coordinator com endpoint HTTP /query com suporte a filtros (ext, domain, before)
  • PostgreSQL (links) com grafo de causalidade topológica entre chunks
  • Ollama local (Mistral, nomic-embed-text) para HyDE e embeddings offline

Restrição central respeitada: O NEXUS não exige reescrita do Hive. Atua como Sidecar de consulta — uma API/CLI que os agentes consomem via MCP ou shell.

A. Integração de Baixo Custo — Quick Wins (Esta Semana) ​

A1. Substituir AGENTS.md bruto por Query Semântica ​

Problema: O Orquestrador injeta ~900 linhas de AGENTS.md em todo prompt. 90% é irrelevante para a task atual.

Solução: Ingerir o AGENTS.md e os ADRs do IntraClinica no NEXUS e usar query pontual no momento da task.

Como funciona hoje no NEXUS:

bash
# O Orquestrador, antes de despachar um worker, faz:
python3 /var/home/bruno/NEXUS/scripts/nexus_context.py \
  --query "Quais são as regras para RLS em módulos SaaS?" \
  --hyde \
  --pack \
  --ext .md

Retorna um Context Pack formatado com apenas os trechos relevantes (L0 bruto + L1 resumo). O worker recebe 200 tokens de contexto cirúrgico em vez de 900 linhas de texto cru.

Como ingerir o corpus do IntraClinica:

bash
# 1. Ingerir regras e ADRs (uma vez, e a cada mudança)
curl -X POST -H "Content-Type: application/json" \
  -d '{"path": "/caminho/para/intraclinica-ng/docs"}' \
  "http://localhost:8000/ingest"

curl -X POST -H "Content-Type: application/json" \
  -d '{"path": "/caminho/para/intraclinica-ng/AGENTS.md"}' \
  "http://localhost:8000/ingest"

> Importante: Arquivos com nexus: ingestible: false no frontmatter são ignorados pelo coordinator (gatekeeping T1). Use isso para excluir artefatos temporários do Hive (spec.md de tasks, por exemplo).

Ganho imediato: O AGENTS.md passa de injeção obrigatória para consulta opcional. O Orquestrador usa 3-5 queries pontuais ao invés de 1 dump monolítico.

A2. MCP Tool para Consulta (Interface para Agentes) ​

Para que os agentes consumam o NEXUS como ferramenta MCP (sem bash manual), criar um wrapper mínimo:

python
# nexus_mcp_tool.py — Wrapper MCP para o axio-coordinator
import httpx

NEXUS_URL = "http://localhost:8000"

def nexus_query(query: str, ext: list[str] | None = None, top_k: int = 5) -&gt; str:
    """
    Tool MCP: Consulta o NEXUS para recuperar contexto arquitetural relevante.
    
    Args:
        query: Pergunta em linguagem natural (ex: "regras de RLS para multi-tenant")
        ext: Filtro de extensão (ex: [".md", ".sql"])
        top_k: Número de resultados (default: 5)
    
    Returns:
        Context Pack formatado para injeção direta no prompt do agente.
    """
    payload = {
        "query": query,
        "top_k": top_k,
        "response_mode": "packs"
    }
    if ext:
        payload["ext"] = ext
    
    resp = httpx.post(f"{NEXUS_URL}/query", json=payload, timeout=30)
    resp.raise_for_status()
    
    data = resp.json()
    packs = data.get("packs", [])
    
    if not packs:
        return "[NEXUS] Nenhum contexto relevante encontrado para esta query."
    
    lines = [f"[NEXUS CONTEXT PACK — Query: '{query}']", ""]
    for pack in packs:
        lines.append(f"=== {pack.get('fs_path', 'unknown')} ===")
        lines.append(pack.get("content", "").strip())
        lines.append("")
    
    return "\n".join(lines)

Integração no opencode.json do IntraClinica:

json
{
  "mcp": {
    "nexus": {
      "type": "stdio",
      "command": "python3",
      "args": ["/var/home/bruno/NEXUS/nexus_mcp_server.py"]
    }
  }
}

Os agentes (Orquestrador e Workers) passam a ter acesso à tool nexus_query sem nenhuma mudança estrutural no fluxo Hive.

A3. Injeção de Decisões Arquiteturais via Grafo Topológico ​

O endpoint /query do coordinator já suporta expansão PPR (Personalized PageRank) de 3 hops via tabela links. Isso significa que uma query sobre "RLS" naturalmente puxa chunks sobre "multi-tenant", "Supabase policies" e "auth middleware" se eles estiverem conectados no grafo.

Como usar o campo domain para separar domínios do IntraClinica:

Ao ingerir, criar collections separadas ou usar filtro ext como proxy de domínio:

  • .sql → regras de banco e RLS
  • .md → ADRs e decisões arquiteturais
  • .ts → padrões de código TypeScript
bash
# Query especializada em regras de banco
python3 nexus_context.py \
  --query "Como implementar isolamento de tenant em queries Supabase?" \
  --hyde --pack --ext .sql .md

B. Integração de Médio Custo — Automação do Fluxo Hive ​

B1. NEXUS como Pré-Seletor de Contexto para Workers ​

Problema atual: O Worker recebe um spec.md genérico e precisa usar grep/find para descobrir quais arquivos ler. Isso consome tokens e frequentemente falha (alucinação por falta de contexto real).

Solução: O Orquestrador, ao criar uma task para um Worker, executa primeiro uma query NEXUS para montar o contexto cirúrgico do worktree.

Protocolo proposto (integração sem reescrita):

bash
# No Orquestrador, antes de despachar task para worker:

# 1. Query baseada na descrição da task
CONTEXT=$(python3 /var/home/bruno/NEXUS/scripts/nexus_context.py \
  --query "$(cat .hive/tasks/task-N/spec.md | head -20)" \
  --hyde --pack --zoom-in --radius 2 \
  --ext .ts .sql .md \
  2&gt;/dev/null)

# 2. Injetar no spec do worker como seção adicional
cat &gt;&gt; .hive/tasks/task-N/spec.md &lt;&lt; EOF

## [NEXUS Context Pack — Pré-Selecionado]
$CONTEXT
EOF

O Worker recebe no spec.md já o contexto relevante dos arquivos. Elimina a necessidade de grep/find no início de cada task.

B2. Code-Awareness: Ingestão Diferencial do Worktree ​

Problema: O NEXUS hoje indexa o repositório inteiro. Workers trabalham em worktrees isolados com mudanças locais não indexadas.

Solução de médio prazo: Adicionar suporte a ingestão por diff antes de despachar o worker.

bash
# Script: nexus_ingest_diff.sh
# Ingere apenas os arquivos modificados no worktree ativo

WORKTREE_PATH=$1
CHANGED_FILES=$(git -C $WORKTREE_PATH diff --name-only HEAD)

for FILE in $CHANGED_FILES; do
  curl -X POST -H "Content-Type: application/json" \
    -d "{\"path\": \"$WORKTREE_PATH/$FILE\"}" \
    "http://localhost:8000/ingest"
done

echo "[NEXUS] Ingestão diferencial concluída: $(echo $CHANGED_FILES | wc -w) arquivos."

Isso garante que o grafo topológico reflita o estado atual do worktree antes da query do worker.

B3. Substituição da Busca de Arquivos por Query Semântica ​

Hoje (Worker com grep):

Worker: "Preciso encontrar onde está implementado o middleware de auth..."
→ grep -r "authMiddleware" src/ → 47 resultados → worker tenta ler todos
→ Saturação de contexto. Alucinação.

Com NEXUS (Worker com query):

Worker usa tool nexus_query("middleware de autenticação JWT IntraClinica")
→ Retorna Context Pack com os 3 arquivos mais relevantes + código L0 exato
→ Worker lê apenas o necessário. Zero alucinação.

O --zoom-in do nexus_context.py retorna o código bruto (L0) dos chunks relevantes, não apenas o resumo L1. O Worker pode editar código real sem inventar variáveis.

C. Integração de Alto Custo — Visão de Longo Prazo ​

C1. NEXUS como CI Semântico (Memória Episódica de Bugs) ​

O problema de longo prazo: Um Worker resolve um bug obscuro na Task 02. Meses depois, o Worker da Task 34 comete o mesmo erro porque não tem acesso à solução anterior.

Solução arquitetural usando T5 (Reflections) + T6 (Tempo):

1. Ao concluir uma task, o Orquestrador grava a solução no NEXUS:

bash
# Após merge bem-sucedido de um worktree:
curl -X POST -H "Content-Type: application/json" \
  -d '{
    "content": "Bug resolvido na Task 02: O middleware RLS não propagava o tenant_id para queries aninhadas. Fix: adicionar SET LOCAL app.tenant_id no início de cada transaction.",
    "source": "hive-task-02",
    "type": "episodic_memory",
    "tags": ["rls", "tenant", "bug", "supabase"]
  }' \
  "http://localhost:8000/reflect"

2. O grafo de causalidade mapeia a solução topologicamente:

A tabela links do PostgreSQL já suporta relações semânticas (causes, depends_on, EVOLVED_FROM). Uma nova relação FIXES pode ser adicionada:

sql
-- Exemplo de link episódico
INSERT INTO links (source_id, target_id, relation_type, metadata)
VALUES (
  '`&lt;chunk_id_do_bug&gt;`',
  '`&lt;chunk_id_da_solucao&gt;`',
  'FIXES',
  '{"task": "task-02", "date": "2026-05-22"}'::jsonb
);

3. O PPR de 3 hops do T4 naturalmente propaga o contexto:

Quando Task 34 trabalha com RLS/tenant_id, a query semântica expande o grafo e encontra a solução da Task 02 como contexto adjacente. O Worker recebe proativamente a memória episódica relevante.

C2. CI Semântico: Gate de Qualidade Arquitetural ​

Visão: Antes de mergear um worktree no draft, o Orquestrador executa um "CI Semântico":

bash
# nexus_semantic_ci.sh — Gate antes do merge
DIFF=$(git -C $WORKTREE show --stat HEAD)
QUERY="Esta mudança viola alguma decisão arquitetural registrada? $DIFF"

CONTEXT=$(python3 nexus_context.py --query "$QUERY" --hyde --pack --ext .md)

# Passar para LLM de validação com o contexto
echo "$CONTEXT" | ollama run mistral "Analise se as mudanças violam ADRs. Responda APROVADO ou REPROVADO com justificativa."

Isso transforma o NEXUS em um guardião arquitetural automático — sem LLM externa, sem custo de API.

C3. Memória de Decisões de Refatoração ​

Durante a refatoração pesada do IntraClinica, cada decisão de design (porque foi escolhido X em vez de Y) deve ser ingerida imediatamente no NEXUS como nota markdown:

markdown
---
nexus: ingestible: true
type: architecture_decision
tags: [auth, supabase, rls]
---
# Decisão: Usar RLS nativo do Supabase em vez de middleware custom

**Contexto:** Task 15 avaliou as duas abordagens.
**Decisão:** RLS nativo porque elimina surface de ataque e simplifica o modelo de permissões.
**Trade-off aceito:** Queries mais complexas para super-admin.

Com isso, o NEXUS acumula memória causal da refatoração. Novos workers 6 meses depois consultam naturalmente por que certas decisões foram tomadas.

Conclusão ​

O NEXUS já é o motor que o IntraClinica precisa. A integração não exige reescrita — exige roteamento. Em vez de injetar contexto bruto nos prompts, os agentes passam a consultar o NEXUS com linguagem natural e recebem context packs cirúrgicos.

O resultado esperado na Fase A: redução de 70-80% no volume de tokens de contexto injetado por task, com aumento de precisão (menos alucinação por excesso de contexto irrelevante).

O NEXUS é a biblioteca perfeitamente indexada. Os agentes Hive são os leitores. O que faltava era o sistema de catalogação — e ele já existe.
