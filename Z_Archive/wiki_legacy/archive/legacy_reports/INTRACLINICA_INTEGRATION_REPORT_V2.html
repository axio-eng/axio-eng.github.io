Relatório Revisado: NEXUS → IntraClinica NG ​

Versão: 2.0 — Baseada em Dados Empíricos
Data: 22 de Maio de 2026
Metodologia: Inspeção direta do banco, git history completo (144 commits), benchmark de throughput real

1. Estado Real da Infraestrutura (O que Funciona de Verdade) ​

Inventário Honesto ​

ComponenteEstado RealDetalhe
Qdrant (T3)✅ Rodando, 132 pontosUp 3 hours. Dados provavelmente de sessão anterior

O Bug do GPU Passthrough ​

O podman inspect axio-llama mostra NVIDIA_VISIBLE_DEVICES=void no ambiente resolvido — o Ollama roda em CPU dentro do container. O Ollama está carregando os modelos na VRAM do host (via acesso direto aos dispositivos /dev/nvidia*), não via CDI/runtime. Isso explica a latência de 0.9s no embedding e o cold-start de 19s: está funcionando, mas sem o caminho otimizado.

2. O Problema Real do Corpus: 85% é Ruído ​

Esta é a descoberta mais importante da análise empírica:

Chunks L0 por categoria (IntraClinica):
  build_artifacts (dist/, dist-test/)  →  4.052 chunks  ← 85% DO CORPUS
  typescript_source (.ts)              →    298 chunks
  sql                                  →    154 chunks
  markdown_docs (.md)                  →    138 chunks
  other                                →     27 chunks
  ─────────────────────────────────────────────────────
  TOTAL                                →  4.669 chunks

Conclusão direta: O t2_worker e t2_bridge nunca rodaram para terminar porque estariam gerando 4.052 abstrações L1 de bundles JavaScript minificados. É trabalho inútil e potencialmente destrutivo para a qualidade semântica do índice.

O corpus útil para a integração com IntraClinica é de ~617 chunks (código-fonte, SQL, docs arquiteturais). Esse é o número real a processar.

3. Os 4 Gaps Críticos (Diagnóstico Confirmado) ​

Gap 1 — T2→T3: 168 L1 gerados, 0 sincronizados ao Qdrant ​

O t2_bridge.py existe e funciona conceitualmente, mas nunca executou com sucesso contra o banco atual. Os 132 pontos no Qdrant são de sessão anterior (provavelmente v3.x), desconectados do PostgreSQL atual. O run_sync_loop.sh (não commitado) está correto mas nunca rodou.

Gap 2 — T4 Grafo: 0 links ​

A tabela links está vazia. O PPR de 3 hops existe no código, mas sem arestas é uma NOP. O script build_lexical_topology.py (Aho-Corasick O(N+M)) que gerava os links foi deletado no cleanup v4.0 (commit ffbb5a6). O build_sparse_topology.py (k-NN via Qdrant) também foi deletado. Ambos existem no git e são recuperáveis.

Gap 3 — Enrichment nunca completou: enriched=false em 100% dos L0 ​

O campo enriched no schema atual nunca é atualizado pelo coordinator Rust — ele gera L1 por outro caminho sem marcar o L0 como processado. O t2_worker.py não tem bug lógico, mas o coordinator e o worker escrevem em rotas paralelas sem coordenação.

Gap 4 — Endpoint /reflect não existe no coordinator ​

A tabela reflections existe. A lógica run_reflect() existe em main.rs (linha 1189), mas é chamada internamente após ingestão — não há rota HTTP exposta. O relatório anterior propôs POST /reflect como API pública; essa rota não existe.

4. O que o Git Revela: A Arqueologia dos Scripts Funcionais ​

O refactor v4.0 (ffbb5a6 — "absolute root-zen cleanup") deletou 38 scripts da pasta scripts/. Esses scripts existem no commit e112a1b (snapshot pre-refactor v3.9) e são recuperáveis em segundos via git show.

Scripts deletados que importam para a integração: ​

5. O Fator GPU: Quanto Tempo Real Leva ​

Com a GTX 1660 SUPER disponível (mesmo via CPU-mode no container, os modelos carregam em VRAM do host):

Benchmark real medido:
  nomic-embed-text (768d):  0.9s por embedding  (warm)
  llama3.2:1b (summarize):  ~2s por chunk        (warm, 30 tokens)

Corpus útil após filtrar build_artifacts:
  617 chunks L0 para enriquecer

Cálculo:
  617 chunks × (0.9s embed + 2s LLM) ≈ 617 × 2.9s ≈ 30 minutos de CPU

Com batch paralelo (BATCH_SIZE=15 do dream_enricher):
  ≈ 5-8 minutos de processamento real

Isso é enriquecimento completo do corpus útil em menos de 10 minutos. A percepção de que seria dias de trabalho era baseada nos 4.739 chunks — incluindo 4.052 de bundles JS que não devem ser processados.

6. Roadmap Revisado: Do Estado Atual ao Funcional ​

Fase 0 — Limpeza + Estabilização (2-4 horas, não dias) ​

Passo 0.1 — Excluir build artifacts do corpus (30 min)

Adicionar regra de path no coordinator ou marcar diretamente:

bash
# Deletar chunks de build artifacts do banco
podman exec axio-postgres psql -U axio -d axiodb -c "
DELETE FROM abstractions 
WHERE blob_id IN (
  SELECT id FROM blobs 
  WHERE fs_path LIKE '%/dist/%' 
     OR fs_path LIKE '%/dist-test/%'
     OR fs_path LIKE '%.min.js'
);
DELETE FROM blobs

Adicionar ao compose.yaml ou ao coordinator uma regra de exclusão por path glob para não reingerir.

Passo 0.3 — Religar o postgres e rodar o pipeline (1-2 horas)

bash
# 1. Subir postgres
podman start axio-postgres

# 2. Rodar enriquecimento sobre corpus filtrado (617 chunks, ~8 min)
python3 scripts/t2_worker.py --limit 700 --radius 2

# 3. Sincronizar L1 ao Qdrant
python3 scripts/t2_bridge.py --limit 300

# 4. Gerar grafo lexical (T4) — Aho-Corasick sobre o corpus limpo
python3 scripts/build_lexical_topology.py

# 5. Gerar grafo de similaridade semântica (T4 complementar)  
python3 scripts/build_sparse_topology.py

# 6. Subir o coordinator Rust
cd services/coordinator && \
DATABASE_URL="postgresql://axio:sovereign_password_change_me@localhost:5432/axiodb" \
QDRANT_URL="http://localhost:6333" \
OLLAMA_URL="http://localhost:8080" \
./target/release/axio-coordinator &

Ao final desse processo, o NEXUS está completamente funcional como descrito no relatório original.

Fase A — Quick Wins (Válidos, mesmos do relatório anterior) ​

Com a Fase 0 concluída, tudo que foi proposto no relatório anterior passa a ser verdadeiro:

  • Query semântica via CLI: nexus_context.py --query "regras RLS SaaS" --hyde --pack funciona
  • MCP Tool: wrapper Python sobre /query do coordinator — 2h de trabalho
  • Substituição do AGENTS.md: ingerir o corpus do IntraClinica e fazer queries pontuais

Estimativa real: 4-6 horas de trabalho técnico para a Fase 0 + 2h para o MCP wrapper.

Fase B — T4 Funcional (Não "Médio Prazo" — Já na Fase 0) ​

Com build_lexical_topology.py recuperado, o T4 fica populado durante a Fase 0. O PPR de 3 hops passa a funcionar imediatamente depois. O que era descrito como "médio prazo" já está resolvido na limpeza inicial.

O diferencial é que o grafo lexical do IntraClinica vai conectar naturalmente arquivos que se referenciam: reception.component.ts → agenda-calendar.component.ts → reception.service.ts. Uma query sobre "agendamento" expande o grafo e puxa todos esses arquivos relacionados.

Fase C — Memória Episódica (Ainda Requer Desenvolvimento) ​

Esta é a única fase que permanece como trabalho novo. O endpoint /reflect não existe na API HTTP. Existem duas opções:

Opção C1 — Ingestão direta (sem desenvolvimento Rust, 1h): Gravar memórias episódicas como arquivos markdown com frontmatter e ingeri-los via /ingest. O coordinator já sabe fazer isso. Perda: sem estrutura de relação FIXES explícita no grafo.

Opção C2 — Adicionar rota /reflect no coordinator (2-3 dias, Rust): Expor a lógica interna run_reflect() como endpoint HTTP. Permite gravação semântica com relação topológica. A lógica já existe — é só uma questão de expor a rota.

Para o IntraClinica em refatoração pesada, Opção C1 é suficiente agora. Documentar decisões como .md no repo e reingeri-las é equivalente em resultado prático.

Conclusão ​

O NEXUS v4.0 é um sistema funcional em estado de hiberna voluntária. O cleanup radical do refactor (commit ffbb5a6) removeu os scripts operacionais corretos — mas eles estão vivos no git. A GPU está disponível e sendo usada. O corpus útil é 7x menor do que parecia (617 vs 4.739 chunks após limpar os bundles de build).

O caminho para o IntraClinica ter um motor de contexto funcional é menos de um dia de trabalho técnico, não duas semanas.
