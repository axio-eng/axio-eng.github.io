🧠 O que o qwen recebe para gerar L2 (Semântico) ​

O prompt enviado ao qwen2.5-coder:1.5b (linhas 659-675 do enrich.rs) contém 3 fontes de dados:

1. L0 — O chunk de código cru (raw) ​

O conteúdo bruto do chunk, truncado em 2000 chars (4000 para SQL):

Analyze the following code chunk (file extension: .ts) and respond with 
ONLY a valid JSON object...

File content:
import { inject } from '@angular/core';
...

2. L1 — Dados estruturais do ast-grep (autoritativos) ​

O JSON do L1 é injetado como contexto com a instrução de não contradizê-lo:

Structural elements from static analysis (authoritative — do not contradict):
{"constructs":{"functions":["getAll","create"],"interfaces":["TemplateEntity"]},"language":"typescript"}

3. 🆕 Vizinhança semântica (sibling files no mesmo diretório) ​

O engine busca até 3 L2 de arquivos vizinhos no mesmo diretório, filtrados por similaridade de cosseno (>0.65):

[NEIGHBORHOOD ECOSYSTEM]
The sibling files in this exact directory have the following semantic profiles:
- template.repository.ts: {"purpose": "This repository handles CRUD..."}

CRITICAL QUESTION: Considering the ecosystem defined by these neighbors, 
what is the specific, exclusive responsibility of THIS new file...?

4. O que o modelo gera (L2) ​

json
{
  "purpose": "Injects necessary services for routing in an Angular application.",
  "classes": [],
  "functions": [{"name": "inject", "params": ["providers: EnvironmentProviders"]}]
}

5. Pós-processamento: L2 é MERGEADO com L1 (ADR-0017) ​

Após o LLM gerar, o engine substitui os campos estruturais pelos dados determinísticos do ast-grep, mantendo só o purpose vindo do LLM:

L2 bruto:  { "purpose": "...", "classes": ["Foo", "Bar"] }  ← LLM pode alucinar
L1 real:   { "constructs": { "classes": ["Foo"] } }          ← ast-grep é preciso
Resultado: { "purpose": "...", "constructs": { "classes": ["Foo"] } }  ← merged

🔗 E o weaver? (conexões entre L2s) ​

Depois que o L2 é vetorizado e enviado ao Qdrant, o weaver (T4) cria 2 tipos de links na tabela agent_links:

LinkComo é criadoTotal hoje
successorConecta cada novo L2 ao L2 anterior (ordem cronológica, arquivos diferentes)engine está processando
semantic_bridgeBusca top-5 vizinhos no Qdrant, conecta se similaridade > thresholdengine está processando

Obs: esses links vão pra agent_links, não pra links. A links só tem evolved_from (versões antigas → novas).

⚡ Sua pergunta: pode conectar num nível mais elevado? ​

Resposta curta: Hoje não. Mas os dados para isso JÁ EXISTEM.

O que o pipeline não faz:

  1. ❌ Síntese por módulo: Não há um passo que colete TODOS os L2 de um módulo (ex: scheduling/) e peça ao modelo "descreva a arquitetura desse módulo"
  2. ❌ Cadeia L0→L1→L2→L3 coesa: O L3 (reflections) opera por blob individual, não por grupo
  3. ❌ Grafos de dependência: O weaver conecta por similaridade vetorial, não por análise de imports/calls

O que você pode fazer:

Opção 1: L3 Modular (mais alto valor imediato) ​

Pegar todos os L2 de um módulo e pedir ao modelo que sintetize:

Colete todos L2 summaries de frontend/src/app/modules/scheduling/
→ Alimente o qwen (ou mistral:7b) com prompt:
  "Synthesize the architecture of this module from its components.
   Identify: bounded contexts, data flow, key patterns, entry points."
→ Gere um L3 module-level, armazene como reflection com abstraction_id=null, domain="scheduling"

Opção 2: Substituir o modelo de L3 ​

As reflections atuais com qwen2.5-coder:1.5b são ruído (modelo pequeno). Com mistral:7b ou qwen2.5-coder:7b, a qualidade sobe drasticamente.

Opção 3: Weaver semântico por módulo ​

Em vez de conectar L2s só por similaridade vetorial, usar o L1 para criar links estruturais reais (ex: arquivo A importa arquivo B → link depends_on).

Quer que eu implemente alguma dessas opções? A Opção 1 (L3 modular) é a de maior impacto imediato — usa os dados já enriquecidos para gerar visão arquitetural de cada módulo do intraclinica-ng.
