Insight: O Grafo como Representação de Schemas ​

Hoje agent_links conecta chunks de texto (L2). Mas a mesma estrutura pode representar schemas inteiros como arestas:

O Que Existe ​

chunk_A ──semantic_bridge──→ chunk_B  (texto relacionado)
chunk_A ──successor────────→ chunk_B  (sequência)

O Que Poderia Existir ​

table_patients ──has_column──→ column_name     (schema DB)
type_User ──────references──→ type_Address     (schema types)
API /patients ──returns─────→ PatientDTO       (schema API)
module_auth ────depends_on──→ module_user      (modular dependency)
file_main.rs ──imports─────→ file_models.rs   (import graph)

Por Que Isso Muda Tudo ​

  1. O T4 deixa de ser "grafo de texto" e vira "grafo do sistema" → arestas nomeadas com semântica fixa (has_column, references, returns) → não depende mais de similaridade vetorial pra existir → navegação estrutural, não apenas semântica

  2. Community detection ganha precisão → clusters não emergem de similaridade de texto, mas de dependência real → "tabelas do domínio paciente" são um cluster porque patient_id referenceia patients → não porque os textos são parecidos

  3. L3 como view do schema → L3 não é "resumo de arquivo" — é "domínio modelado por arestas"

    python
    cluster = community_detect(agent_links, filter="has_column|references|depends_on")
L3 = synthesize(cluster)  # "Domínio Paciente: tabelas, APIs, tipos"

  4. Zoom estrutural

    depth=0: domínio (cluster de schemas)
depth=1: entidades do domínio (tabelas, types, APIs)
depth=2: campos, colunas, endpoints específicos
depth=3: código L0 correspondente

O Pipeline Novo ​

AST (L1)           ──extract──→ call_graph edges   ──┐
Blobs (L0)         ──parse───→ schema edges         ──┤──→ agent_links
Chunks (L2)        ──embed───→ semantic_bridge      ──┘
                                                       │
                                                       ▼
                                              Community Detection
                                                       │
                                                       ▼
                                              L3 por domínio
                                              (view do cluster)

O Que É Priorityo Agora ​

  1. Extrair schemas do código (parser de tabelas, types, APIs, imports)
  2. Inserir como agent_links com criteria semântica (has_column, references, etc.)
  3. Rodar community detection sobre o grafo enriquecido
  4. Validar se os clusters correspondem a domínios reais
  5. L3 emerge naturalmente — sem síntese forçada por blob
