NEXUS Runtime Canônico — Contrato Operacional ​

Propósito ​

Este documento define o contrato operacional do runtime NEXUS vigente. Ele descreve o fluxo de dados, as responsabilidades de cada substrato e as três primitivas MCP que compõem o núcleo da interação agente-sistema. Não substitui docs/T4_GRAPH_NORMS.md (navegação) nem AGENTS.md (protocolo do agente), mas as amarra num fluxo único.

As Três Primitivas de Runtime ​

nexus_query — Entrada por Similaridade Semântica (T3) ​

A porta de entrada do runtime. Uma query em linguagem natural é embedada (nomic-embed-text, 768d) e busca no Qdrant coleção nexus_ghost.

  • Alvo: L2 (sumário semântico) via similaridade coseno
  • Triangulação: Quando triangulate=true, o coordinator cruza os resultados L2 com os equivalentes L1 (AST) e L0 (código-fonte) do mesmo blob, retornando um pack tri-nível
  • Disclaimer: a busca é por proximidade semântica, não por matching estrutural. Para busca estrutural, use collection='nexus_ast'
nexus_query("auth guard redirect", triangulate=true)
  → pack: L2 semântico + L1 AST + L0 código-fonte

nexus_adjacent — Expansão Topológica (T4) ​

A partir de um node_id (UUID de qualquer nó no grafo T4), navega pelas arestas do grafo multi-relacional.

  • Subir: adjacent(chunk) → part_of → file node
  • Descer: adjacent(file_node) → inbound part_of → lista de chunks
  • Cruzar: adjacent(file_node) → depends_on:imports → arquivos importados
  • Projeção tri-axial: criteria × file_type × project — cada chamada fatia o grafo num ângulo específico

Para a especificação completa dos tipos de aresta, padrões de navegação e eixos de projeção, consulte docs/T4_GRAPH_NORMS.md.

nexus_link — Cristalização Estigmérgica ​

Registra uma descoberta não-óbvia entre dois nós no grafo T4 como uma aresta semantic_bridge com decaimento temporal (expires_at).

  • Quando usar: após completar uma investigação multi-hop em que o agente descobre uma relação que não está capturada por depends_on:imports
  • Comportamento: arestas sem tráfego futuro evaporam (decaimento ativo). A cristalização precisa ser confirmada por uso — é um contrato fraco, intencionalmente
  • Contrapartida: para fatos estruturais (ex: "arquivo A importa B"), não use nexus_link — isso é função do pipeline L1
nexus_query("...") → node_id A
  → nexus_adjacent(A) → node_id B
  → nexus_link(A, B, criteria="semantic_bridge", rationale="...")

Query Path (Fluxo de Execução) ​

Toda consulta do runtime segue o mesmo pipeline de 5 etapas:

1. QUERY É EMBEDADA
   └─ nomic-embed-text (768d) → vetor denso

2. T3 LOCALIZA SEMENTES
   └─ Qdrant, coleção nexus_ghost → top-k L2 por similaridade coseno
   └─ (opcional) collection='nexus_ast' para busca estrutural L1

3. T4 EXPANDE VIA ADJACENT
   └─ coordinator recebe chunk_ids → nexus_adjacent() em cada semente
   └─ sobe (part_of) → file node → cruza (depends_on:imports) → desce (part_of)
   └─ subgrafo ativo é montado na janela da consulta

4. T2 HIDRATA CONTEÚDO
   └─ PostgreSQL: abstractions, blobs, agent_links, reflections
   └─ hidrata: caminho, versão, chunk_index, metadados, arestas vivas

5. COORDINATOR MATERIALIZA PACKS
   └─ estrutura final: { L2 semântico, L1 AST, L0 fonte, metadados T4 }
   └─ retorno para o agente via MCP

Responsabilidades de Cada Substrato ​

SubstratoCamadaFunção
QdrantT3Entrada por proximidade vetorial. Coleções nexus_ghost (L0+L2) e nexus_ast (L1). Busca por similaridade coseno.
PostgreSQLT2 + T4Âncora canônica de identidade (abstractions, blobs), versionamento (superseded_by_id), arestas topológicas (agent_links), e memória derivada (reflections, architectural_insights).
FilesystemT1Fonte material reconstruível. T1 vence em caso de divergência entre substratos (reingestão a partir do fs).

Pipeline de Enriquecimento ​

A ingestão de novos artefatos segue uma esteira sequencial determinística:

file watcher (inotify/poll)
  → L0 chunking (~2000 chars, overlap)
    → L1 AST (ast-grep, blob completo → JSON determinístico)
      → L2 enrichment (qwen2.5-coder:1.5b, L0 + L1 → sumário semântico)
        → T3 sync (Qdrant: nexus_ghost + nexus_ast)
          → T4 edge computation (agent_links: part_of, depends_on:imports,
               semantic_bridge, contradicts)

L3 não é mais produzido por blob. A síntese de domínio emerge de forma orgânica via community detection no grafo T4 (agent_links), utilizando algoritmos como LPA ou walktrap. Cada cluster detectado = candidato a domínio L3. A geração pode ser fixa (armazenada) ou on-the-fly (sintetizada no momento da consulta). Consulte docs/adr/0015-zoom-stateful-protocol.md para a justificativa completa.

Regras Operacionais ​

  1. T1 vence em divergência: Se o conteúdo no filesystem discordar do que está no Postgres ou Qdrant, o filesystem é a fonte da verdade. Reingestão total a partir de T1 é o mecanismo de correção.
  2. Reingestão total é mecanismo aceitável: Mudança de modelo de embedding, topologia ou geometria pode exigir TRUNCATE blobs CASCADE, limpeza das coleções Qdrant e reingestão completa.
  3. Sem suporte prioritário a legados: O runtime atual não mantém compatibilidade retroativa com formatos de resposta anteriores. Agentes devem usar as primitivas MCP — não endpoints REST legados.
  4. Stateless por chamada: Cada tool MCP é independente. Não há estado de sessão entre chamadas — o grafo T4 e o Qdrant são a memória.
  5. Evaporação estigmérgica: arestas criadas por nexus_link têm expires_at. Se nenhum agente transitar por elas, evaporam. A memória do runtime é viva, não cumulativa.
