NEXUS v5 Hybrid AST Pipeline ​

Overview: Beyond Naive Top-K ​

Most RAG systems operate on a simple contract: chunk a document, embed each chunk into a vector database, and retrieve the top-K nearest neighbors at query time. This works for plain text but fails for code.

A codebase is not a bag of tokens. It has:

  • Syntax structure (classes inside modules, functions inside classes, inheritance chains)
  • Version history (the same file changes over time, old chunks become stale)
  • Cross-file relationships (imports, type references, interface implementations)
  • Semantic intent (what a function does is different from how it's structured)

Naive top-K treats every chunk as an independent island. It cannot distinguish between src/main.rs and target/debug/main.rs. It has no concept of "this chunk supersedes that chunk." It returns 8 random snippets and calls it context.

NEXUS v5 solves this with a 4-layer Hybrid AST Pipeline that separates concerns into distinct processing stages:

LayerWhat it capturesHow its producedDeterminism
L0Raw verbatim code sliceschunk_text() splits blob at paragraph boundariesDeterministic
L1Static AST structure (classes, functions, interfaces, etc.)ast-grep binary runs on the full blobDeterministic (zero LLM)
L2Semantic summary (intent, purpose of each chunk)qwen2.5-coder:1.5b LLM fed L0 + L1 contextNon-deterministic (LLM)
L3Long-horizon architectural insightsqwen2.5-coder:1.5b reads L2 summaries periodicallyNon-deterministic (LLM)

The pipeline uses two complementary storage engines:

  • Qdrant (T3) = The GPS: Fluid semantic proximity. Entry point for all agent queries. Houses two collections: nexus_ghost (L0+L2) and nexus_ast (L1).
  • PostgreSQL (T2/T4) = The Exact Map: Rigid deterministic storage. Blobs, abstraction hierarchy, temporal links (superseded_by_id), topological graph (hilbert_bridge), and reflections.

Architecture Overview ​

Phase 1: Synchronous Ingestion ​

Phase 1 runs synchronously in the file watcher's receive loop. When notify detects a file modification, creation, or rename event, the path is sent through a channel to ingest_file() (main.rs:516).

Key function: chunk_text() (main.rs:880) ​

The chunking strategy is paragraph-aware, not fixed-size sliding window:

rust
fn chunk_text(text: &str, size: usize) -&gt; Vec`&lt;String&gt;` {
    let mut chunks = Vec::new();
    let mut current = String::new();

    for line in text.lines() {
        if line.trim().is_empty() && !current.is_empty() {
            if current.len() &gt; 20 { chunks.push(current.clone()); }
            current.clear();
        } else {
            if !current.is_empty() { current.push('\n'); }
            current.push_str(line);
            if current.len() &gt;= size {
                chunks.push(current.clone());
                current.clear();
            }
        }
    }
    if current.len() &gt; 20 { chunks.push(current); }
    chunks
}

Rules:

  • Paragraphs are separated by blank lines (empty/whitespace-only lines)
  • A chunk starts a new paragraph boundary and accumulates lines until either the next blank line or the CHUNK_SIZE limit (2500 chars, main.rs:35)
  • Chunks under 20 characters are discarded (noise filter)
  • This preserves logical grouping (a function's docblock + signature stays together)

L0 Qdrant Upsert: Dense + Sparse Vectors ​

Each L0 chunk generates a hybrid vector (main.rs:730-742):

  • text_dense: 768d embedding from nomic-embed-text via Ollama (main.rs:681)
  • text_sparse: BM25-style sparse vector computed from the chunk's word frequencies (main.rs:709-728), enabling full-text keyword boost via Qdrant's sparse vector search

The payload includes fs_path, ext (file extension), chunk_index, is_current, ingested_at, version, and the raw content for RRF full-text boost (main.rs:697-707).

Versioning Strategy (Strategy C, main.rs:507-514) ​

NEXUS does not use in-place UPDATEs. Each content change creates a new blob row:

Partial UNIQUE INDEX on (fs_path) WHERE is_current = TRUE
  • Insert new row with version = old.version + 1
  • Set prev_blob_id = old.id (linked list for version traversal)
  • Set old row's is_current = FALSE
  • Temporal T6 links (superseded_by_id) connect old abstractions to new ones at matching chunk_index levels (main.rs:828-851)

This makes version history queryable via /versions or /timeline endpoints (main.rs:1940-2187).

Phase 2: Async Enrichment ​

Phase 2 runs asynchronously via run_enrichment() (main.rs:2683), which is triggered either by:

  • Calling POST /enrich (main.rs:2466)
  • The watcher debounce timer (5s loop, triggers enrichment 10s after last file event, main.rs:2261-2274)
  • The daemon check on NEXUS_ENRICH_SCHEDULE env (if set, not shown but referenced in handle_pipeline_status)

Phase 2a: L1 AST via ast-grep (Deterministic) ​

CRITICAL: L1 operates on the full blob content, not chunks (main.rs:2686-2688):

rust
// CRITICAL: Queries blobs directly, NOT L0 chunks. Chunks have broken syntax
// that causes ast-grep parsing failures. Full blob content ensures valid AST.

The function astgrep_extract() (main.rs:2604) does:

  1. Write the full file content to a temporary file
  2. Call ast-grep binary with language-specific patterns
  3. Parse JSON output, extract names of structural constructs
  4. Clean up temp files
  5. Return a JSON object like: {"language": "python", "constructs": {"classes": ["Foo"], "functions": ["bar", "__init__"]}}

Patterns are defined per language in sg_patterns_for_lang() (main.rs:2521). For example, Rust patterns include structs, enums, functions, traits, type aliases, constants, statics, and macros. Python patterns include classes, functions, and async functions.

The result is stored in abstractions at level = 1 with chunk_index = -1 (meaning blob-level, not chunk-level) (main.rs:2739-2742).

Phase 2b: L2 Semantic via Ollama (LLM) ​

L2 enrichment creates a semantic summary for each L0 chunk by feeding it to qwen2.5-coder:1.5b via Ollama's /api/generate (main.rs:2773-2870).

The prompt asks for a structured JSON with fields: classes, functions, interfaces, imports, exports, decorators (main.rs:2793-2794). The LLM response is stored in abstractions at level = 2, maintaining the same blob_id and chunk_index as the source L0 chunk.

Jobs are processed in parallel with buffer_unordered(2) (main.rs:2867) to avoid overwhelming the GPU.

Phase 2c: Embed and Upsert to Qdrant ​

Any L1 or L2 abstraction without a vector_id is embedded via nomic-embed-text and upserted to the appropriate Qdrant collection (main.rs:2872-2958):

  • L1 (AST) → nexus_ast collection — using qdrant_collection(1) which returns "nexus_ast" (main.rs:26-33)
  • L2 (Semantic) → nexus_ghost collection — using qdrant_collection(2) which returns "nexus_ghost" (default)

The vector_id is then back-filled in the abstractions row and enriched = true is set (main.rs:2941).

Phase 3: L3 Reflections ​

When the cumulative count of newly inserted chunks reaches REFLECT_THRESHOLD (50, main.rs:40), the system spawns run_reflect() (main.rs:1307) as a background task.

run_reflect() queries up to 20 L0 abstractions that have no existing reflection, then sends each to qwen2.5-coder:1.5b with the prompt: "Derive a single concise insight (1-2 sentences) from this text." The insights are stored in the reflections table (main.rs:1347-1354).

This is the system's attempt at long-horizon architectural understanding — not just indexing code, but deriving durable insights about the codebase over time.

Critical Rule: AST on BLOB, Not on CHUNK ​

This is the single most important architectural invariant in the v5 pipeline.

Never generate AST from an L0 chunk. Here's why.

An L0 chunk is a verbatim slice of a file, split at paragraph boundaries (blank lines). A file like:

typescript
import { Component } from '@angular/core';

@Component({
  selector: 'app-root',
  template: '`&lt;h1&gt;`Hello&lt;/h1&gt;',
})
export class AppComponent {
  title = 'nexus';

  constructor() {
    console.log('constructed');
  }
}

When chunked, might produce slices like:

Chunk 0:
import { Component } from '@angular/core';

@Component({
  selector: 'app-root',
Chunk 1:
})
export class AppComponent {
  title = 'nexus';

  constructor() {

Each of these chunks has broken syntax. Tree-sitter and ast-grep require a complete, syntactically valid parse tree. Feeding them a chunk that starts with }) and ends with constructor() { will produce parse errors, wrong AST, or empty results.

The rule (from AGENTS.md section 3, and enforced in main.rs:2686-2688):

> The coordinator passes the entire blob (full file content) to ast-grep -> produces 1 JSON per file -> stored as L1.

The run_enrichment() function queries the blobs table directly (not the abstractions table at level 0) when looking for files that need L1 AST enrichment (main.rs:2695-2708):

sql
SELECT b.id, b.fs_path
FROM blobs b
WHERE b.is_current = TRUE
  AND EXISTS (SELECT 1 FROM abstractions a WHERE a.blob_id = b.id AND a.level = 0)
  AND NOT EXISTS (SELECT 1 FROM abstractions l1 WHERE l1.blob_id = b.id AND l1.level = 1)
LIMIT 25

The raw_content column of blobs contains the full, unbroken file text (main.rs:2713):

rust
let raw_content: Option`&lt;String&gt;` = sqlx::query_scalar(
    "SELECT raw_content FROM blobs WHERE id = $1"
)

The qdrant_collection() function (main.rs:26-33) routes:

rust
fn qdrant_collection(level: i32) -&gt; &'static str {
    match level {
        1 =&gt; AST_COLLECTION,      // "nexus_ast"
        _ =&gt; DEFAULT_COLLECTION,  // "nexus_ghost"
    }
}

A third collection, nexus_reflections, stores L3 embeddings but is not yet wired into the query path (it appears in handle_pipeline_status at main.rs:1439).

Storage Duality ​

  • Qdrant is the fast semantic index. All queries enter through Qdrant k-NN search. It knows proximity, not identity.
  • PostgreSQL is the canonical store. It knows which blob version is current, which abstraction supersedes which, and the topological graph (hilbert_bridge) for the PPR walk.

When a query arrives (main.rs:903-1178):

  1. Qdrant returns top-100 vector IDs with cosine scores
  2. IDs are resolved to abstractions.id via PostgreSQL (main.rs:977-991)
  3. T4 PPR walk expands the seed set across hilbert_bridge edges in PostgreSQL (main.rs:1002-1060)
  4. Final results are re-ranked, filtered by extension/temporal/domain constraints, and truncated to top_k (main.rs:1062-1177)

Extensions that return None from ext_to_sg_lang() will cause astgrep_extract() to return an error (main.rs:2611-2612), and L1 generation is skipped for that file. L0 chunking and embedding still proceed normally for any file matching NexusConfig.extensions.

Exclusion and Filtering ​

The configurable NexusConfig (main.rs:44-72) controls two filters:

  • extensions: Which file extensions to ingest (default: md, rs, py, txt, toml, yaml, yml, sql)
  • excluded_dirs: Which directory names to skip (runtime data, build artifacts, agent artifacts: axio_data, postgres_v2, qdrant, ollama, target, node_modules, site-packages, venv, __pycache__, .git, .angular, dist, .hive, .worktrees, .opencode, etc.)

The is_excluded() function (main.rs:421-459) also checks YAML frontmatter for nexus: ingestible: false (fast-scan: only reads first 1KB of each file). This is how documentation files in docs/ exclude themselves from the vector space to prevent semantic loops (AGENTS.md section 3).

The is_supported() function (main.rs:410-416) checks whether the file extension is in the configured list.

The File Watcher ​

The watcher in watch_fs() (main.rs:2337) uses notify's RecommendedWatcher for cross-platform file system events. Key behavior:

  • Modify/Create events: Path sent to ingest channel (main.rs:2390-2395)
  • Remove events: Marks the blob as is_current = FALSE without ingestion (main.rs:2383-2388)
  • Rename events: Marks the old path as not current, links the new blob's prev_blob_id to the old blob's ID (main.rs:2366-2381)
  • Gitignore-aware: Respects .gitignore patterns via the ignore crate (main.rs:2341-2355)

A 5-second debounce loop (main.rs:2261-2274) triggers run_enrichment() if a file event occurred within the last 10 seconds, ensuring enrichment runs soon after batch file changes settle.

The initial_scan() function (main.rs:2292-2335) runs at startup using WalkBuilder to queue all existing files into the ingest channel, then marks stale fs_nodes as requiring re-ingestion.

Query Path ​

For completeness, the query flow (main.rs:903-1178):

  1. User query is embedded via nomic-embed-text (or custom model) (main.rs:915)
  2. Qdrant k-NN search in specified collection (main.rs:951-961)
  3. Vector IDs resolved to abstractions.id via PostgreSQL (main.rs:977-991)
  4. T4 PPR Walk: Up to 3 hops across hilbert_bridge edges, propagating scores with alpha=0.85 decay (main.rs:1002-1060)
  5. T2 re-hydration: Fetch full content, path, version, chunk_index (main.rs:1086-1156)
  6. Filtering by temporal (before), domain (cluster_id), and extension (ext) (main.rs:1104-1142)
  7. Truncation to top_k (main.rs:1176)
  8. Optional context pack assembly (main.rs:1180-1263): For each seed, fetch neighboring chunks within context_radius to produce coherent context windows
