MCP Interface and Agent Use Cases ​

Overview: NEXUS as Cortex Central ​

NEXUS is not just a code indexer. It is the Cortex Central for a cluster of small, specialized AI agents that share a persistent semantic memory. Each agent is lightweight and focused, holding a minimal context window. Instead of loading the entire codebase (which causes semantic drift and exhausts context budgets), agents query NEXUS via the Model Context Protocol (MCP) to retrieve exactly the code, structure, and decisions they need.

The NEXUS pipeline continuously transforms raw source code through four layers:

LayerNameWhat it storesHow it's produced
L0Raw ChunksVerbatim code slices (~2000 chars)File watcher splits blobs
L1Static ASTStructural map of each full fileast-grep (deterministic, zero LLM)
L2Semantic SummaryIntent/purpose of each chunkqwen2.5-coder:1.5b on L0 + L1
L3ReflectionsLong-horizon architectural insightsqwen2.5-coder:1.5b on L2 summaries

These four layers define the memory contract that any agent can access through MCP. The query path is: agent question -> Qdrant vector search (T3) -> PPR topological diffusion (T4) -> PostgreSQL hydration -> Context Pack returned to agent. (nexus_context.py:79-124, nexus_context.py:156-220, nexus_context.py:223-273)

Each agent is small and focused -- minimal context window, no brute-force reading. Everything goes through NEXUS. Agents can also write back to NEXUS via nexus_link to record architectural relationships and via nexus_reflections to read high-level design decisions. (AGENTS.md:225-247)

MCP Tools Reference ​

The NEXUS MCP server (nexus_mcp_server.py) exposes seven tools. It runs in two modes: stdio (invoked on-demand by OpenCode) and HTTP/SSE (long-running server on port 7474 for external agents). (nexus_mcp_server.py:179-196, nexus_mcp_server.py:604-664)

The Agent Interaction Pattern ​

Every agent that enters the NEXUS ecosystem follows the same disciplined protocol:

The pattern is always: question -> nexus_query -> grounded understanding -> implement -> optionally record insight. No agent ever brute-force reads the codebase. No agent ever guesses about file contents. (AGENTS.md:8-10)

Use Case 1: Angular Specialist Agent ​

Scenario ​

An agent needs to understand how the scheduling module manages component state with Angular Signals, then implement a new feature.

Step 1: Module Topography ​

nexus_map(module="scheduling", format="text")

Returns the structural map of the scheduling module: which services exist, which signals they expose, what components use them, and the inject() dependency tree.

Step 2: Semantic Grounding ​

nexus_query(
  query="como o sistema gerencia o estado e signals no modulo de agendamento",
  zoom_in=true,
  hyde=true
)

The engine applies HyDE (Hypothetical Document Embeddings) to expand the colloquial Portuguese query into technical keywords like signal, WritableSignal, computed, inject, toObservable. (nexus_context.py:40-62) It then searches Qdrant's nexus_ghost collection with level=1 filter (L1 abstractions only), runs PPR diffusion through the topological graph, and hydrates the matched chunks with their L0 raw code neighbors. (nexus_context.py:79-123, nexus_context.py:156-220)

The returned Context Pack contains:

  • L1 summaries describing each file's purpose and signals
  • L0 raw code for the exact signal declarations, services, and templates

Step 3: Implementation ​

The agent now has precise knowledge of:

  • How WritableSignal is used for state
  • How computed() derives values
  • How effect() handles side effects
  • The exact module file structure

It implements the new feature following the exact same patterns, using real variable names and function signatures from the context pack, not invented ones.

Use Case 2: Security Auditor Agent ​

Scenario ​

An agent must audit authentication and authorization across the stack, checking for RLS policies, injection risks, and proper RBAC implementation.

Step 1: Cross-Layer Query ​

nexus_query(
  query="autenticacao JWT, RLS policies, role-based access",
  extensions=[".ts", ".sql", ".prisma"],
  zoom_in=true,
  hyde=true
)

Filters by extensions to search only TypeScript, SQL, and Prisma files. (nexus_mcp_server.py:38-43) The HyDE system translates the conceptual query into technical search terms like jwt, signIn, createClient, row_level_security, policy for, authenticated role.

Step 2: Check Architectural Reflections ​

nexus_reflections(query="auth security decisions", limit=10)

Returns up to 10 L3 reflections -- coordinator observations about past auth-related decisions, known vulnerabilities, or architectural invariants written after periodic analysis of L2 summaries.

Step 3: Epistemological Audit ​

The nexus_context.py engine also supports an Epistemological Audit mode using Qdrant's Recommendation API. (nexus_context.py:126-153) The auditor can search for contradictions:

python
# Conceptual: find code that matches "RLS is enabled" but contradicts "insecure data access"
cmd_contradict("row level security policies ativas", "dados acessiveis sem autenticacao")

This uses Qdrant's positive/negative vector parameters to find code semantically close to one concept but far from another -- effectively finding friction zones.

Step 4: Architecture Map ​

nexus_map(project="intraclinica-ng")

Returns the full project file listing plus abstraction drift metrics (AGENTS.md:37-42). High drift files (where code has diverged from its high-level design) are flagged automatically.

Use Case 3: Architecture Review Agent (Oracle) ​

Scenario ​

An Oracle agent needs to evaluate whether a proposed change aligns with existing architectural decisions, then record a new decision.

Step 1: Read Architectural Reflections ​

nexus_reflections(limit=10)

Returns timestamped L3 architectural insights generated by the coordinator. (nexus_mcp_server.py:528-557) These are high-level design decisions and structural invariants discovered through periodic analysis of L2 summaries.

Step 2: Drift Analysis ​

nexus_query(
  query="quality map for modules with high abstraction drift",
  extensions=[".ts"]
)

The --quality-map flag in nexus_context.py (AGENTS.md:37-42) returns files ranked by abstraction_drift -- the cosine/L2 distance between L1 (high-level design) and L0 (implementation) embeddings. High drift signals that implementation has strayed from the intended design.

Step 3: Decision Recording ​

After completing the review, the Oracle writes back to NEXUS via nexus_link to crystalize discovered relationships between code chunks. This creates persistent edges in the stigmergic graph that future agents can navigate via nexus_adjacent.

Step 4: Structural Tour ​

nexus_map(format="json")

With format="json", returns a machine-parseable structure of the entire project: all services, stores, guards, models, interfaces, signals, and their inject() dependencies.

Use Case 4: Debug Session Agent ​

Scenario ​

A bug report arrives: "Form submission in the clinical-practice module is silently failing." A debugging agent needs to trace through the layers.

Step 1: Semantic Seed ​

nexus_query(
  query="form submission error handling clinical-practice salvamento silencioso",
  zoom_in=true,
  hyde=true,
  extensions=[".ts"]
)

The query engine finds relevant L1 abstractions (summaries of form handlers, HTTP services, error interceptors) and returns them with their surrounding L0 chunks via the zoom_in / context_radius mechanism. (nexus_mcp_server.py:258-259, nexus_context.py:253-267)

Step 2: Trace Through Layers ​

The L2 semantic summaries (generated by qwen2.5-coder:1.5b) describe the intent of each chunk -- not just what the code does, but why it exists. The L0 raw chunks provide the exact code. Together they let the agent trace:

L2 summary -&gt; "This interceptor catches HTTP 400 errors and logs them"
L0 code    -&gt; "return throwError(() =&gt; new AppError('validation', err))"

Step 3: Check Architectural Context ​

nexus_reflections(limit=10, domain="clinical-practice")

Returns high-level L3 architectural insights relevant to the clinical-practice module, revealing any known design decisions, invariants, or past issues.

Step 4: Context Restoration ​

The agent can reconstruct the full debugging timeline by combining:

  1. Current L0/L2 context from nexus_query
  2. L3 architectural context from nexus_reflections
  3. Graph-based dependency exploration via nexus_adjacent

The Mixture of Agents Architecture ​

The endgame for NEXUS is not a single monolithic agent. It is a cluster of specialized agents -- each tiny, each focused, all sharing NEXUS as their collective memory. (AGENTS.md:225-247)

Key Principles ​

  • Each agent is small and focused -- minimal context window, no need to load the full codebase.
  • No brute-force reading -- every agent queries NEXUS first, always. (AGENTS.md:8-10)
  • Shared memory is persistent -- what one agent learns (via nexus_link or the automatic L2 enrichment pipeline) becomes available to all.
  • The pipeline is automatic -- L0->L1->L2->L3 runs continuously in the background via the Rust coordinator, fed by file watchers. (AGENTS.md:71-74)
  • Agents can modify the pipeline -- via nexus_admin(action='enrich') to trigger re-indexing, and by writing architectural relationships via nexus_link.
