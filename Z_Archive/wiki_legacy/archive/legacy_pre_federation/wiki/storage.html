NEXUS v5 Storage Architecture ​

Overview: Why Two Storage Engines ​

NEXUS treats memory as a navigable space, not a bag of vectors. No single storage engine can provide both fluid semantic proximity and rigid deterministic identity at the same time. That is why the system uses two complementary databases, each optimized for a different cognitive role.

Qdrant is the GPS. It knows where things are relative to each other in semantic space. Given a query vector, it returns the nearest neighbors by cosine distance. It does not know what a file is, which version is current, or how chunks connect to form a document. It only knows proximity.

PostgreSQL is the Exact Map. It knows the absolute position of every piece of data: file paths, version numbers, chunk indices, timestamps, and the topological graph that connects abstractions through hilbert_bridge and supersedes relations. It cannot guess what you mean. It can tell you exactly what exists.

You navigate by GPS (Qdrant) to find promising coordinates, then switch to the Exact Map (PostgreSQL) to understand what lives there, hydrate the content, and navigate structured relationships.

This separation mirrors the distinction between semantic retrieval (what is related to this concept?) and structural recall (what are the chunks of this file, in order, at this version?). A single database optimized for both always degrades one dimension.

PostgreSQL (T2-T4) = The Exact Map: Rigid, deterministic."

Table Details ​

fs_nodes — Filesystem Truth ​

The outermost ring of the storage model. Every file NEXUS tracks gets a row here. The content_hash field enables fast staleness detection: when a file changes on disk, its hash diverges from the stored value, and the watcher flags it as pending.

code/migrations/001_init.sql:4-10
main.rs:546-557 — INSERT with ON CONFLICT (path) DO UPDATE

Key columns:

  • path (TEXT PK): Relative path from the NEXUS data root.
  • content_hash (TEXT): SHA-256 of the full file content at last scan.
  • sync_status (TEXT): pending (needs reingestion) or synced (up to date).
  • modified_at (TIMESTAMPTZ): Filesystem mtime, used as fallback for content_date.
  • last_ingested_at (TIMESTAMPTZ): Updated every successful ingestion.

Status filter index (003_temporal_indexes.sql:5-7):

sql
CREATE INDEX IF NOT EXISTS idx_fs_nodes_sync_status
  ON fs_nodes (sync_status)
  WHERE sync_status != 'synced';

blobs — Versioned Content Anchors ​

Each file on disk maps to one or more blobs. When a file changes, the system creates a new blob row with version = old + 1 and marks the old row as is_current = FALSE. This is the immutability-by-version strategy.

main.rs:509-514 — Strategy C: versioned blobs with partial unique index
main.rs:596-616 — Version increment on content change

Key columns:

  • id (UUID PK): Auto-generated primary key.
  • fs_path (TEXT): Relative path. A partial unique index idx_blobs_fs_path_current ensures only one current version per path (workspace/database/migrations/002_blob_versioning.sql:29-31).
  • raw_content (TEXT): The full text of the file at ingestion time. Used by ast-grep for L1 generation (must be complete, not chunked).
  • source_hash (TEXT): SHA-256 of raw_content. Used for cache invalidation without re-reading chunks.
  • version (INT): 1-based version counter, incremented each time the file changes.
  • prev_blob_id (UUID): Self-referential FK pointing to the previous version. Enables linear version traversal.
  • is_current (BOOL): Exactly one row per fs_path has is_current = TRUE. The rest form a historical chain.
  • content_date (TIMESTAMPTZ): Canonical temporal anchor. Resolved at ingestion time from: (1) YAML frontmatter created_at or date, (2) filesystem mtime, (3) current time.
main.rs:369-402 — extract_content_date() with 3-fallback chain

abstractions — The Ladder of Abstraction ​

The central table of the storage model. Every chunk at every level of the pipeline lives here. L0 rows are created during ingest_file(), L1 and L2 rows during enrichment (run_enrichment()), and they share the same schema with different level values.

code/migrations/001_init.sql:21-43 — Original abstractions schema
main.rs:764-777 — L0 INSERT in ingest_file()
main.rs:2845-2849 — L1/L2 INSERT in run_enrichment()

Key columns:

  • id (UUID PK): Used as the canonical identifier throughout the system. Links in T4, Qdrant points, and context packs all reference abstractions.id.
  • blob_id (UUID FK): Owning blob. All chunks of the same file (same version) share the same blob_id.
  • level (INT): 0 for raw chunks, 1 for AST, 2 for semantic summary, 3 for synthesized, 4 for axiomatic.
  • chunk_index (INT): Position within the blob, starting at 0. For L1 (blob-level AST), this is -1.
  • content (TEXT): For L0, verbatim slice of the original file (~2000 chars). For L1, JSON output from ast-grep. For L2, LLM-generated summary.
  • vector_id (UUID): Maps to the Qdrant point UUID. NULL until enrichment embeds and upserts the content.
  • superseded_by_id (UUID): T6 temporal chain. When a file is reingested, old L0 chunks that match by blob_id + chunk_index + level get this column set to the new abstraction's id.
  • ingested_at (TIMESTAMPTZ): When this row was inserted. Used for temporal filtering in queries.
  • enriched (BOOL): True after the content has been embedded and upserted to Qdrant.
main.rs:828-851 — T6 temporal linking via superseded_by_id
main.rs:2941 — UPDATE abstractions SET vector_id = $1, enriched = true WHERE id = $2

links — Topological Graph (T4) ​

Abstractions are connected through bidirectional edges in the links table. Two relation types are active in production:

  • hilbert_bridge: Semantic edges computed by the topology pipeline. These form a navigable graph that the PPR walker traverses during T-RAG queries.
  • supersedes: Temporal edges. Currently disabled since v3.8-beta because pairwise O(N^2) generation caused combinatorial explosion (>400k rows).
main.rs:802-824 — supersedes disabled, commented out with explanation

A third relation type, evolved_from, is registered by a PostgreSQL trigger when superseded_by_id is set (workspace/database/migrations/005_time_aware_memory_and_nuance.sql:35-83). The trigger also rewires existing links from the old abstraction to the new one.

Key columns:

  • source_id, target_id (UUID FK): The two abstraction endpoints.
  • relation (TEXT): One of hilbert_bridge, supersedes, or evolved_from.
  • weight (FLOAT): Base edge weight, typically 1.0.
  • stress (FLOAT): Hamiltonian tension metric computed by hilbert_stress() in the topology pipeline.
  • nuance_weight (FLOAT8): Free-form gradation weight for contextual relevance scoring.

PPR edge traversal in the query path:

main.rs:1002-1059 — PPR walk: up to 3 hops, decay factor PPR_ALPHA=0.85, min score PPR_MIN_SCORE=0.01
main.rs:1014-1022 — SQL with temporal decay: EXP(-0.05 * EXTRACT(EPOCH FROM ...))

reflections — L3 Insights ​

Periodically, the coordinator feeds unreflected L0 chunks to qwen2.5-coder:1.5b and stores the generated insights:

The Two-Vector Design in nexus_ghost ​

L0 chunks are stored with both a dense vector and a sparse vector in the same point. The dense vector (text_dense, 768d from nomic-embed-text) captures semantic meaning. The sparse vector (text_sparse) encodes word frequencies as hashed indices, enabling a form of full-text retrieval inside the vector store.

main.rs:697-742 — Qdrant upsert payload with dual vector structure
main.rs:709-728 — Sparse index construction: word hashing + frequency scoring

This hybrid approach means a single Qdrant search can combine semantic similarity (dense) with keyword matching (sparse) without needing a separate full-text index.

The Query Path ​

How a search request flows through both storage systems.

Step-by-step ​

  1. Embed (main.rs:461-505): The query text is sent to Ollama's /api/embeddings endpoint using nomic-embed-text (768 dimensions). Retries up to 3 times with exponential backoff (2s, 4s, 8s).

  2. Qdrant k-NN search (main.rs:918-972): The dense vector is sent to the specified collection (nexus_ghost by default). Up to 100 seed points are returned. If exact_node is specified, a payload filter is applied on fs_path (ghost) or node_name (ast).

  3. Vector ID to Abstraction ID (main.rs:974-991): Qdrant returns point UUIDs (which match abstractions.vector_id). These are mapped to canonical abstractions.id values through a PG query. This remapping is necessary because T4 topological links reference abstractions.id, not Qdrant point IDs.

  4. PPR Walk on T4 (main.rs:1002-1059): If the target collection is nexus_ghost (and not an exact search), the coordinator runs up to 3 Personal PageRank hops over the active links graph. Each hop propagates a fraction (PPR_ALPHA = 0.85) of the Qdrant score to neighboring abstractions, weighted by edge stress and attenuated by temporal decay (EXP(-0.05 * days_since_content_date)). Nodes below PPR_MIN_SCORE = 0.01 are pruned.

  5. T2 Hydration and Filtering (main.rs:1062-1177): Ranked abstraction IDs are joined with blobs to fetch full content, fs_path, chunk_index, and version. Three optional filters are applied:

    • Temporal (before): Only abstractions with content_date or ingested_at before a given ISO-8601 timestamp.
    • Extension (ext): Only blobs whose fs_path matches .rs, .py, etc.
    • Domain (domain): Only abstractions belonging to a specific K-Means cluster.

  6. Context Pack Materialization (main.rs:1180-1263): When response_mode = "packs" (default), each seed is expanded to a contiguous window of chunks (context_radius = 1 by default) within the same blob. The result is a ContextPack with a center chunk, score, and neighbor chunks — providing document-aware context rather than isolated fragments.

Query Metrics ​

Each query returns three metrics:

  • edge_flow: Total propagated score through the T4 graph during PPR walk. High values indicate a topologically dense region.
  • region_density: Sum of all scores in the expanded result set. Measures how much semantic mass exists near the query.
  • query_drift: Sum of scores acquired only through graph traversal (not from Qdrant seeds). Indicates how much the search expanded beyond direct vector similarity.
main.rs:1073-1084 — QueryMetrics computation

Temporal Links (T6) ​

When a file is reingested, the old and new versions are linked through two mechanisms:

Blob-level Chain ​

workspace/database/migrations/002_blob_versioning.sql — version + prev_blob_id + is_current

Each new blob row increments the version and sets prev_blob_id to the old blob's ID. The old blob gets is_current = FALSE. A partial unique index ensures exactly one current blob per path.

Abstraction-level Supersession ​

Within each blob version, individual chunks at the same position and level are linked through superseded_by_id:

main.rs:828-851 — UPDATE abstractions SET superseded_by_id = a_new.id
                     FROM abstractions a_new
                     WHERE a_new.blob_id = $1         -- new blob
                       AND a_old.blob_id = $2         -- old blob
                       AND a_new.chunk_index = a_old.chunk_index
                       AND a_new.level = a_old.level
                       AND a_old.superseded_by_id IS NULL

This produces a chain of vector_id values: old chunk points to new chunk at the same position. The UPDATE only fires when all four conditions match (new blob, old blob, same chunk index, same level) and the old chunk hasn't already been superseded.

Trigger-based Graph Healing ​

When superseded_by_id is set, a PostgreSQL trigger (trg_heal_topological_links) automatically rewires all T4 links that pointed to the old abstraction to now point to the new one. It also inserts an evolved_from link from new to old, preserving the lineage:

workspace/database/migrations/005_time_aware_memory_and_nuance.sql:35-83

This means the topological graph stays coherent across reingestions without manual link maintenance.

The hilbert_bridge — Topological Graph Navigation ​

The hilbert_bridge table (actually the links table with relation = 'hilbert_bridge') is a topological expansion layer. It connects abstractions that are semantically related but may not be nearest neighbors in Qdrant space.

Why It Exists ​

Vector search finds the top-k closest points. But the most useful context for an LLM agent often includes:

  • Related functions in the same module
  • Documentation that references the same concepts
  • Configuration files that affect the same component

These may not land in the top-k of a Qdrant search. The PPR walk on the hilbert_bridge graph absorbs this ambient context.

How Edges Are Used ​

During query, each PPR hop propagates scores outward:

main.rs:1041-1055 — Score propagation:
  flow = (source_score * PPR_ALPHA * edge_weight) / source_degree
  next[target] += flow

The PPR_ALPHA = 0.85 parameter controls how much energy leaves the original seeds each hop. After 3 hops, only (0.15)^3 ≈ 0.3% of the original score remains, so expansion is shallow but enough to cross file boundaries.

Temporal Decay on Edges ​

Edge weight is multiplied by a temporal decay factor:

sql
EXP(-0.05 * EXTRACT(EPOCH FROM (now() - COALESCE(b.content_date, b.last_ingested_at))) / 86400.0)

(main.rs:1016-1017)

This gives approximately exp(-0.05 * days) weight to old content. After 30 days, the factor is ~0.22, effectively deprioritizing historical edges. The intent is to let the graph naturally favor current code during expansion.

Data Flow Summary ​

Write Path ​

  1. A file change is detected by the filesystem watcher (notify crate, main.rs:2357-2401).
  2. is_excluded() checks the path against config rules and YAML frontmatter nexus.ingestible flag (main.rs:421-459).
  3. The file is hashed (SHA-256) and compared to the current blob's source_hash. If unchanged, ingestion is skipped (main.rs:579-589).
  4. If changed, the old blob is marked is_current = FALSE and a new blob row is created with version = old + 1 (main.rs:591-641).
  5. The raw content is split into ~2000 character chunks by chunk_text() (main.rs:880-899). Blank-line boundaries are respected.
  6. Each chunk is:
    • Embedded via Ollama to a 768d dense vector + sparse word hash indices
    • Upserted to nexus_ghost in Qdrant with payload { fs_path, ext, chunk_index, is_current, version, ingested_at }
    • Inserted into abstractions with level = 0, linked to the new blob
  7. After ingestion, T6 temporal links are set: old chunks at matching (blob_id, chunk_index) point to new chunks via superseded_by_id (main.rs:828-851).
  8. The enrichment pipeline runs asynchronously: L1 (ast-grep on full blob), L2 (LLM summary on each chunk), then embedding + upsert to the appropriate Qdrant collection.

Read Path ​

  1. The query text is embedded to a 768d vector via nomic-embed-text.
  2. Qdrant returns up to 100 nearest neighbors from the target collection.
  3. Vector IDs are mapped to abstractions.id in PostgreSQL.
  4. The PPR walker traverses the T4 hilbert_bridge graph for up to 3 hops, propagating scores with temporal decay.
  5. Ranked results are hydrated from PostgreSQL with optional temporal, extension, and domain filters.
  6. Context packs are assembled by expanding each seed chunk with its neighbors within the same blob.
