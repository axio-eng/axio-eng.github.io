Difusão Semântica de Tempo Limitado em T4 ​

Status: Decisão de runtime
Data: 2026-03-24
Documento companheiro: GEOMETRIA_DE_FLUXO_T3.md, FORMAL_ANALYSIS_HAMILTONIAN.md

Propósito ​

Este documento responde a uma pergunta precisa: por que o sistema NEXUS não sofre de explosão combinatória de caminhos ao fazer difusão semântica sobre o grafo T4?

A GEOMETRIA_DE_FLUXO_T3.md define o que é difusão em termos conceituais — o campo induzido pela query, o custo de caminho J(γ), a integral de linha sobre a malha. A FORMAL_ANALYSIS_HAMILTONIAN.md define a função Hamiltoniana de stress que pesa cada aresta. Este documento explica o mecanismo de tempo limitado que torna a difusão computacionalmente viável.

A resposta não é "porque limitamos o número de hops". A resposta é que três mecanismos ortogonais atuam juntos para garantir que o espaço de busca permaneça restrito mesmo em grafos grandes.

1. A semântica de w_q(e): condutância, não trabalho ​

O documento GEOMETRIA_DE_FLUXO_T3.md define w_q(e) como o peso contextual de uma aresta sob a query. Na formulação contínua:

J(γ) = ∫_γ F_q(x) ds

Na formulação discreta:

J(γ) = Σ_{e ∈ γ} w_q(e)

Aqui há uma distinção conceitual fundamental que a notação não torna explícita. w_q(e) não é trabalho. Não é energia acumulada. É condutância — a capacidade de uma aresta transferir relevância de um nó a outro.

A distinção importa porque:

  • Trabalho (ou energia) é conservado. O que é consumido em um passo não reaparece.
  • Condutância é um coeficiente de transferência. Ela determina quanto do score de um nó flui para o nó vizinho a cada passo de Random Walk.

No código (main.rs:825-827), w é calculado como:

sql
(COALESCE(l.stress, 1.0) * 
EXP(-0.05 * EXTRACT(EPOCH FROM (now() - COALESCE(b.content_date, b.last_ingested_at))) / 86400.0)
)::float8 as w

Ou seja:

w = stress × exp(-0.05 · Δt / 86400)

Onde Δt é o tempo transcorrido desde a última modificação do conteúdo. A aresta perde condutância exponencialmente com a idade. Isso reflete uma escolha de design: conexões antigas transferem menos relevância, porque a geometria semântica pode ter mudado.

O stress da aresta (do Hamiltonian) é a condutância base. O fator de decaimento temporal é um atenuador. O produto é a condutância efetiva no momento atual.

2. Fluxo versus trabalho: a equação de transferência ​

A GEOMETRIA_DE_FLUXO_T3.md usa J(γ) para denotar o trabalho acumulado ao longo de um caminho. Isso é consistente com a física: trabalho é força vezes deslocamento, acumulado ao longo de uma trajetória.

Mas o runtime atual não calcula J(γ) explicitamente. Ele computa fluxo — a taxa de transferência de score por passo do Random Walk.

No código (main.rs:855-860):

rust
let flow = (s * PPR_ALPHA * w) / s_deg;
*next.entry(tgt.clone()).or_insert(0.0) += flow;

Onde:

  • s é o score (score de PageRank personalizado) no nó fonte
  • PPR_ALPHA é o fator de teleportação (0.85)
  • w é a condutância da aresta
  • s_deg é o grau do nó fonte (número de arestas saindo dele)

A fórmula de fluxo é:

flow = s · α · w / deg

Isso é a lei de Ohm para difusão semântica: a corrente (flow) é igual ao potencial (s), vezes a condutância (w), dividido pela resistência (deg). O grau do nó atua como resistência — quanto mais arestas saindo de um nó, menor a fração de score que vai para cada vizinho específico.

A key insight: o fluxo não se acumula ao longo do caminho. Cada passo do Random Walk é independente. O score no nó t após k hops é a soma de todos os caminhos de comprimento k que terminam em t, mas o sistema nunca enumera esses caminhos explicitamente. Ele mantém apenas os scores nodais e aplica a matriz de transição estocástica.

Isto contrasta com J(γ) da GEOMETRIA_DE_FLUXO_T3.md, que é o trabalho total ao longo de um caminho específico. J(γ) seria a quantidade relevante se estivéssemos resolvendo um problema de otimização de trajetória (e.g., "qual é o caminho de menor custo?"). Mas para difusão de relevance, o que importa é a distribuição estacionário do Random Walk — não o custo de caminhos individuais.

3. Inércia: como o score entra no modelo ​

Na física clássica, inércia é a tendência de um corpo em movimento a permanecer em movimento. No contexto de difusão semântica, o score s no nó atua como energia cinética — a "quantidade de movimento" que o nó tem para transferir aos vizinhos.

No Hamiltonian formal (FORMAL_ANALYSIS_HAMILTONIAN.md), a energia de uma aresta é:

H(s, t) = max(0, ||V_t - T ⊙ V_s||² / N - δ) · √(M_s · M_t) · w

O Hamiltonian é uma função de energia potencial (stress). Mas o que faz a difusão andar não é o stress diretamente — é o score.

O score s começa como a distribuição de partida (das sementes do Qdrant). A cada hop, o score se propaga pela regra:

s_{k+1}(t) = Σ_{s → t} s_k(s) · α · w(s,t) / deg(s)

O fator (1 - α) da teleportação garante que uma fração do score original (das sementes) seja reinjetada a cada passo, impedindo a distribuição de colapsar para zero ou divergir.

A interpretação física: o score é a energia cinética离散 no nó. A condutância w é a "seção transversal" da aresta. O grau deg é a "resistência total do nó". A constante α = 0.85 é o fator de inércia — 85% da energia cinética do nó é redistribuída a cada passo; 15% é reinjeção direta das sementes.

4. Os três mecanismos ortogonais que previnem explosão combinatória ​

Esta é a questão central do documento. A formulação teórica de difusão sobre um grafo poderia, em princípio, explorar todos os caminhos possíveis. Em um grafo denso, o número de caminhos de comprimento k cresce exponencialmente com k. Por que isso não acontece no NEXUS?

Mecanismo 1: Horizonte de hops limitado ​

O primeiro mecanismo é o mais direto: o sistema executa exatamente PPR_MAX_HOPS = 3 iterações de Random Walk (main.rs:812).

rust
const PPR_MAX_HOPS: usize = 3;
const PPR_ALPHA: f64 = 0.85;
const PPR_MIN_SCORE: f64 = 0.01;

for _hop in 0..PPR_MAX_HOPS {

Três hops significam que qualquer caminho considerado tem no máximo 3 arestas. O número de caminhos de comprimento ≤ 3 a partir de n sementes é O(n · d²) onde d é o grau máximo do grafo — polynomial, não exponencial.

Por que 3? Porque a partir de 4 hops, a condutância efetiva cai abaixo de qualquer limiar útil. A cada hop, o score é multiplicado por α = 0.85 e dividido pelo grau. Após 3 hops, o score acumulado de qualquer caminho já é menor que 0.01 — o limiar de ruído (PPR_MIN_SCORE).

Mecanismo 2: Atenuação exponencial temporal ​

O segundo mecanismo está no fator exp(-0.05 · Δt / 86400) em main.rs:826.

Para uma aresta com Δt = 1 dia:

exp(-0.05 / 86400) ≈ exp(-5.79 · 10⁻⁷) ≈ 0.9999994

Para Δt = 30 dias:

exp(-0.05 · 30 / 86400) ≈ exp(-1.737 · 10⁻²) ≈ 0.9828

Para Δt = 90 dias:

exp(-0.05 · 90 / 86400) ≈ exp(-5.21 · 10⁻²) ≈ 0.949

O decaimento é muito lento no curto prazo. Mas o efeito cumulativo ao longo de hops é significativo: cada aresta com mais de ~20 dias de idade reduz a condutância em ~5% por hop.

O ponto não é apenas o decaimento absoluto, mas o filtro dinâmico: arestas antigas transferem tão pouca condutância que seus vizinhos não sobrevivem ao limiar PPR_MIN_SCORE = 0.01 após 3 hops. O grafo efetivo de difusão é muito menor que o grafo completo.

Mecanismo 3: Retenção por limiar de score ​

O terceiro mecanismo está em main.rs:866:

rust
next.retain(|_, v| *v &gt;= PPR_MIN_SCORE);

Após cada hop, todos os nós com score menor que 0.01 são descartados da distribuição. Isso tem dois efeitos:

  1. Poda do espaço de estados: o número de nós ativos após cada hop é bounded por n / 0.01 onde n é a soma dos scores iniciais — ou seja, no máximo ~100 nós por semente ativa.
  2. Eliminação de paths de baixa relevância: qualquer caminho que acumule menos que 0.01 de score total é cortado. Como o score de um caminho de comprimento k é aproximadamente s₀ · (α · w_avg)^k, um caminho precisa de w_avg suficientemente alta para sobreviver 3 hops.

Juntos, esses três mecanismos garantem que o espaço de busca após 3 hops seja O(n · d²) com um fator de poda que elimina nós de baixa relevância em cada camada. O sistema nunca precisa enumerar todos os caminhos — ele mantém apenas a distribuição de score por nó.

5. Por que a dimensão infinita de Hilbert nunca é computada explicitamente ​

A GEOMETRIA_DE_FLUXO_T3.md menciona que T3 funciona como espaço expressivo de incidência local. O FORMAL_ANALYSIS_HAMILTONIAN.md define o Hamiltonian sobre vetores de dimensão N = 768 (para nomic-embed-text).

Uma pergunta que poderia surgir: o sistema precisa computar algo no espaço de Hilbert de dimensão infinita?

A resposta é não, e a razão está no fato de que o sistema opera sobre representações discretas.

O Hamiltonian H(V_s, V_t, T, δ, w, ℓ_s, ℓ_t) é definido sobre vetores V_s, V_t ∈ ℝ^N. Cada nó do grafo tem um embedding de dimensão fixa N = 768. A difusão opera sobre os scores nodais — escalares — não sobre os embeddings.

O espaço de Hilbert de dimensão infinita seria relevante se estivéssemos:

  • Fazendo kernel methods sobre o espaço de features contínuas
  • Resolvendo problemas de otimização variacional no espaço de funções
  • ComputandoSimilaridade em espaços com product dot infinito

Mas o NEXUS não faz nada disso. Ele:

  1. Reduz a query a um conjunto de sementes (via T3, que usa dot product no espaço de embeddings)
  2. Propaga scores escalares sobre o grafo (via T4, que usa Random Walk com condutância)
  3. Hidrata conteúdo dos nós selecionados (via T2)

A dimensão N = 768 aparece apenas no cálculo do stress da aresta (Hamiltonian). Uma vez que o stress é computado e persistido como peso da aresta, a dimensão do embedding não é mais usada na difusão.

Em resumo: o sistema discretiza o problema no nível do grafo. Os embeddings de alta dimensionalidade são usados apenas para computar pesos de aresta uma única vez (offline, no momento da criação da aresta). A difusão é completamente tractable porque opera sobre escalares.

6. Horizonte limitado versus convergência ao equilíbrio ​

Há uma distinção importante entre o que o sistema faz e o que aconteceria se deixássemos o Random Walk rodar até o equilíbrio.

O que o sistema faz: horizonte limitado ​

O sistema executa PPR_MAX_HOPS = 3 iterações fixas. Isso é um horizonte limitado (bounded horizon). A distribuição de score após 3 hops é:

s_3 = P³ · s_0

Onde P é a matriz de transição do Random Walk e s_0 é a distribuição inicial (sementes do Qdrant).

O horizonte limitado tem uma propriedade importante: ele preserva localidade temporal. Nós que são próximos às sementes em termos de hops recebem mais score do que nós que são distantes. Isso é exatamente o que queremos para montagem de contexto: o contexto relevante tende a estar perto das sementes semânticas.

O que aconteceria com convergência de equilíbrio ​

Se deixássemos o Random Walk rodar até a distribuição estacionária (k → ∞), teríamos:

s* = lim_{k→∞} P^k · s_0 = π

Onde π é o stationary distribution — que para um grafo não-direcionado é proporcional ao grau do nó. Nesse caso, todos os nós seriam ranqueados principalmente pelo seu grau, não pela sua relevância para a query.

A FORMAL_ANALYSIS_HAMILTONIAN.md discute a convergência de PPR em grafos direcionados (§4.1). Em grafos direcionados, a distribuição estacionária pode não ser única se houver múltiplas componentes fortemente conectadas. A convergência de PPR no grafo NEXUS direcionado é uma questão em aberto (Conjectura 9.1 no documento).

Por que horizonte limitado é a escolha certa ​

Para montagem de contexto, não queremos convergência de equilíbrio. Queremos:

  1. Localidade: contexto que está topologicamente próximo das sementes
  2. Recência: arestas antigas transferem menos score (via decaimento temporal)
  3. Direcionalidade: o grafo é dirigido, então o score flui ao longo de relações específicas

Um horizonte fixo de 3 hops garante que o contexto gerado seja uma vizinhança expandida das sementes, não o resultado de um processo de relaxamento global. O limite de 3 não é arbitrário — ele reflete o ponto onde o score acumulado de qualquer caminho adicional cai abaixo do limiar de ruído.

Contraste com PageRank tradicional: PageRank tradicional sem restart (α = 0) converge para a distribuição estacionária, que é dominada pelo grau. PPR com restart (α &gt; 0) converge para uma distribuição que é uma interpolação entre a distribuição estacionária e a distribuição de restart. O sistema NEXUS nem siquiera converge — ele para após um número fixo de hops.

7. Síntese: a arquitetura de difusão limitada ​

O mecanismo de difusão do T4 pode ser descrito em uma frase: um Random Walk com 3 hops, onde cada aresta tem condutância dependente de idade e o score de cada nó é redistribuído proporcionalmente ao grau, com poda de limiar.

As três perguntas que este documento responde são:

  1. O que w_q(e) representa? Condutância, não trabalho. É a capacidade de transferência de relevância, não energia acumulada.

  2. Por que não há explosão combinatória? Três mecanismos ortogonais: horizonte fixo de 3 hops, atenuação temporal exponencial, e poda por limiar de score. Juntos, eles garantem que o espaço de busca efectivo seja O(n · d²) com fator de poda por camada.

  3. Por que a dimensão de Hilbert não importa? Porque o sistema discretiza no nível do grafo. Embeddings de alta dimensão são usados apenas para computar pesos de aresta offline. A difusão opera sobre escalares.

Este documento complementa a GEOMETRIA_DE_FLUXO_T3.md (que define o que é difusão conceitualmente) e a FORMAL_ANALYSIS_HAMILTONIAN.md (que define como o stress é calculado). Juntos, eles explicam: o que é difusão, como ela é pesada, e por que ela termina em tempo finito e útil.
