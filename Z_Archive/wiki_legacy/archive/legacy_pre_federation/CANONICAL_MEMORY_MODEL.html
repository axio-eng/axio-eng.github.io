Modelo Canônico de Memória Operacional ​

Propósito ​

NEXUS é memória externa operacional para agentes e LLMs. Diferente de cache ou armazenamento frio, ela sustenta continuidade cognitiva: o agente retoma linhas de raciocínio, recupera decisões passadas e navega entre código, documentação e contexto abstrato sem depender exclusivamente da janela de contexto nativa.

A metáfora central é a de um organismo vivo: a memória não é um depósito flat de chunks, mas um sistema com camadas de rigidez distintas — esqueleto, carne e cartilagem — que se coordenam para viabilizar navegação precisa e adaptativa.

O Triângulo T3 → T4 → L3 ​

A recuperação no NEXUS não é single-stage. Três mecanismos cooperam em sequência, cada um com um papel distinto:

CamadaSubstratoFunçãoMetáfora
T3Qdrant (vetorial)Detecção por proximidade vetorial — encontra sementes semanticamente próximasGPS: mostra onde você está no espaço semântico
T4Grafo multi-relacionalExpansão topológica — caminha por dependências, similaridades e hierarquias entre arquivosMapa de ruas: mostra rotas entre bairros do código
L3Síntese emergenteSíntese de domínio que emerge de comunidades no T4, não de blob individual indexadoBairro: clusters de nós que juntos formam um conceito coerente

Papel de cada vértice ​

T3 (GPS) é a porta de entrada. O agente chega com uma pergunta em linguagem natural; T3 a embeda e localiza os pontos mais próximos no espaço vetorial. É rápido, impreciso por construção e serve para encontrar onde começar, não para entregar a resposta final.

T4 (Mapa de ruas) é o expansor de contexto. A partir das sementes do T3, o T4 caminha pelo grafo multi-relacional: sobe via part_of para o arquivo, cruza via depends_on:imports ou semantic_bridge para arquivos relacionados, desce para chunks vizinhos. Cada aresta é uma projeção diferente do sistema — estrutural, semântica ou hierárquica.

L3 (Bairro) não é um level fixo nem um blob pré-indexado. Diferente de L0/L1/L2 (níveis de chunk atômicos), L3 emerge on-the-fly por detecção de comunidades no T4. Um agente não "consulta L3" — ele navega pelo T4 até que um cluster semanticamente coerente se revele. A síntese de domínio é resultado da topologia, não de um documento pré-fabricado. L3 substitui o antigo conceito de "abstração panorâmica" por um mecanismo emergente.

Exemplo concreto do triângulo ​

Um agente perguntando "como o sistema valida permissões de acesso?" dispara o triângulo completo:

  1. T3 embeda a pergunta e encontra chunks sobre AuthGuard, RBAC e PermissionService — sementes próximas no espaço vetorial.
  2. T4 recebe os node_ids desses chunks, sobe para os file nodes correspondentes, cruza por depends_on:imports para descobrir que AuthGuard usa TokenService e UserRepository, e desce para os chunks de implementação destes últimos.
  3. L3 emerge da comunidade formada por esses nós — não como um documento separado, mas como o cluster topológico {AuthGuard, TokenService, UserRepository, PermissionService} que o agente reconhece como "sistema de autorização".

Sem T3, o agente não teria ponto de partida. Sem T4, ele teria só chunks isolados sem conexão. Sem L3, ele não saberia que aqueles arquivos formam um domínio coerente.

Por que três camadas e não duas ​

T3 e T4 resolvem problemas ortogonais. Proximidade vetorial (T3) não captura hierarquia estrutural — dois arquivos podem ser semanticamente próximos mas não ter relação de dependência. Topologia (T4) não captura similaridade difusa — dois arquivos podem depender do mesmo módulo sem serem semanticamente parecidos. L3 emerge justamente da interseção: comunidades densas no T4 que também são próximas no espaço T3. O triângulo não é redundância; é coordenação de mecanismos complementares.

Anatomia da Memória ​

O grafo T4 — e por extensão toda a memória operacional — tem três camadas de rigidez, como um organismo:

Tabela de Camadas ​

CamadaMetáforaArestasMutabilidadeFunção
RígidaEsqueletodepends_on:imports, part_ofFato — imutávelRelações estruturais que independem de modelo: A importa B no código-fonte, chunk pertence a um arquivo
SemânticaCarnesemantic_bridge, contradictsAproximação — mutávelSimilaridades e conflitos detectados por embedding; mudam com threshold, modelo ou pipeline
ClassificaçãoCartilagemfile_type, projectMetadado — classifica os nósTags que permitem fatiar o grafo por formato, projeto ou tipo sem alterar a conectividade

Por que a anatomia importa para a memória operacional ​

O agente não navega todo o grafo a cada consulta — isso seria computacionalmente inviável e cognitivamente disperso. A anatomia define prioridade de navegação:

  1. Esqueleto primeiro. Arestas estruturais (depends_on:imports, part_of) são verdadeiras independente de modelo de embedding. O agente confia cegamente nelas para subir à árvore de arquivos, descer a chunks e cruzar dependências reais de importação. Essa é a âncora que impede deriva semântica.

  2. Carne depois. Arestas semânticas (semantic_bridge, contradicts) são dicas exploratórias. O agente usa para descobrir território desconhecido — arquivos que "parecem com" o atual — mas sabe que podem conter falsos positivos. Carne é hipótese, não fato.

  3. Cartilagem sempre. file_type e project filtram o escopo da navegação. O agente fatia o grafo por esses eixos para ignorar formatos irrelevantes (ex.: excluir JSON e YAML ao buscar regras de negócio) e focar no projeto correto.

Essa priorização impede que o agente se perca em similaridades superficiais antes de estabelecer a âncora estrutural. É o que diferencia navegação guiada por topologia de busca puramente vetorial.

Como a anatomia se manifesta no runtime ​

No fluxo de recuperação concreto, as três camadas aparecem em momentos diferentes:

  • Antes da consulta (indexação): o pipeline de enriquecimento classifica cada nó com file_type e project (cartilagem), extrai depends_on:imports do código-fonte (esqueleto), e calcula semantic_bridge e contradicts entre pares de arquivos (carne). As camadas são pré-computadas, não inferidas em tempo de query.

  • Durante a expansão T4: o agente decide quais arestas seguir com base na confiança que cada camada merece. Para navegação estrutural (entender a arquitetura), segue só depends_on:imports — ignora a carne. Para descoberta exploratória (encontrar arquivos similares a um dado módulo), ativa semantic_bridge — mas trata os resultados como hipóteses.

  • Na montagem do context pack: a cartilagem (file_type) ajuda o agente a priorizar formatos. Uma query sobre regras de negócio em TypeScript pode pular arquivos JSON, YAML ou markdown, mesmo que eles apareçam na expansão. O agente fatia o grafo pelos eixos de projeção no momento da hidratação.

Essa separação entre o que é fato estrutural, o que é aproximação semântica e o que é classificação contingente permite que o agente navegue com granularidade variável — precisa quando precisa, difusa quando está explorando.

Capacidades Desejadas ​

  1. Recuperar casos antigos relevantes — o agente reencontra decisões, código e discussões anteriores mesmo depois de múltiplas sessões, sem depender de memória de trabalho do LLM.
  2. Relacionar código + decisão + contexto — uma única consulta conecta implementação (L1), intenção semântica (L2) e fundamentação arquitetural (reflections T5), formando um pacote coerente.
  3. Sustentar continuidade além da janela nativa — a memória externa faz o papel que o contexto LLM não consegue: reter longas linhas de raciocínio entre interações, cruzando sessões e pausas.
  4. Transferência entre contextos — solução encontrada em um módulo é navegável para outro módulo análogo via expansão T4, sem necessidade de reconsulta ou re-embedagem.

O Que Não É o Objetivo ​

  • Não é ontologia rígida. O NEXUS não impõe o que "merece" virar conhecimento. A ingestão permanece ampla — código, notas, markdowns, decisões e rascunhos entram sem curadoria prévia. Camadas derivadas servem para navegação, não para policiar o corpus.
  • Não é busca flat. O sucesso não depende exclusivamente de top_k no Qdrant. A expansão T4 e a síntese emergente L3 são o que diferenciam o sistema de um retriever vetorial commodity.
  • Não é verdade absoluta. As camadas semânticas (carne) são aproximações. O agente precisa calibrar o nível de confiança conforme a camada — esqueleto é fato, carne é hipótese.

Runtime Atual ​

O fluxo de recuperação ativo segue três estágios:

  1. Entrada T3. A consulta é embedada e busca sementes por similaridade vetorial no Qdrant.
  2. Expansão T4. A partir das sementes, o grafo multi-relacional expande o contexto: sobe via part_of para o nó arquivo, cruza via depends_on ou semantic_bridge para arquivos relacionados, desce para chunks vizinhos.
  3. Hidratação T2. O Postgres fornece conteúdo bruto, metadados, caminho e versão para montar os context packs entregues ao agente.

O resultado é um context pack: uma janela coerente de memória que combina código-fonte (L0), estrutura AST (L1), sumário semântico (L2) e relacionamentos topológicos (T4) — não uma lista flat de chunks.
