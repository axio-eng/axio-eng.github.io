ADR-0017: L2 Structure Inheritance from L1 — Static Analysis as Ground Truth ​

Status ​

Accepted

Context ​

The NEXUS pipeline has 4 abstraction levels:

LevelWhatGeneratorDeterministic?
L0Raw code chunksFile watcherYes (verbatim)
L1Static AST (classes, functions, interfaces, etc.)ast-grepYes
L2Semantic summary (purpose, intent)LLM (qwen2.5-coder:1.5b)No
L3Architectural reflectionsLLM (qwen2.5-coder:1.5b)No

The L2 prompt (at main.rs lines 3211–3227) asks the LLM to generate both structural fields (classes, functions, interfaces, imports, exports, signals, injects for TypeScript; tables, functions, policies, triggers for SQL) AND a semantic purpose field. The L1 AST JSON is injected as "authoritative — do not contradict" context, but the LLM is still asked to reproduce structural data.

This design has proven problematic:

  1. LLM hallucinates structural data. Confirmed cases: L2 inventing AudioBase64Parser class, SpinnerComponent in a data-table file, enum with icons in a register component, and a fabricated FhirImplementation class. These are not minor errors — they are structurally impossible constructs that pollute the semantic vector space.

  2. Redundant computation. ast-grep already extracts every structural construct deterministically in L1 (lines 2730–2801 of main.rs). Asking the LLM to re-extract the same information wastes tokens, increases latency (~200ms/chunk with qwen2.5-coder:1.5b), and produces worse results.

  3. L2 content in Qdrant nexus_ghost is structurally incomplete. After the LLM generates L2 JSON, it's stored raw in abstractions.content and upserted to Qdrant as-is (Phase 2, lines 3360–3368). The L1 structural fields are not carried over — they live in a separate collection (nexus_ast). This means a semantic search on nexus_ghost cannot directly filter by structural properties like "files that export AuthGuard."

  4. The semantic drift gatekeeper (lines 3166–3198, documented in ADR-0014) validates neighborhood semantic relevance but does not validate structural fields against L1. There is zero cross-validation between L2's structural claims and L1's ground truth anywhere in the pipeline.

Decision Drivers ​

  • Determinism is more reliable than LLM output for structural facts. ast-grep is a parser — it cannot hallucinate.
  • Token budget is constrained on local GPU (GTX 1660 Super, 6GB VRAM). Each redundant structural field in the L2 prompt costs tokens that could be used for better semantic analysis.
  • Qdrant payloads should be self-contained. An L2 point in nexus_ghost should carry its structural fingerprint without requiring a cross-collection join to nexus_ast.
  • L1 is already stored per blob (chunk_index=-1), while L2 is per chunk (chunk_index=0..N). The structural data applies to all chunks of the same blob.
  • Backward compatibility with existing data in Qdrant and PostgreSQL must be maintained.

Considered Options ​

Option 1: Post-generation L1×L2 validation (detect-only) ​

  • Pros: Minimal code change, non-destructive
  • Cons: Detects but does not prevent hallucinations. Still stores corrupted data in Qdrant. Requires a separate validation pipeline. Half-measure.

Option 2: L2 inherits L1 structure at upsert time (selected) ​

  • Pros: Eliminates hallucination at the root. L2 content in Qdrant becomes complete (purpose + L1 structures). No separate validation pipeline needed. Simpler than rewriting the prompt.
  • Cons: Requires merge logic at Phase 2. Prompt still wastes tokens on structural fields until next prompt revision. Merge logic must handle edge cases (no L1 exists for a blob, L2 is for a blob that has no ast-grep support, e.g., Markdown).

Option 3: Rewrite L2 prompt to ask only for "purpose" + merge at Phase 2 ​

  • Pros: Cleanest separation of concerns. Zero structural extraction by LLM. Shortest prompt.
  • Cons: More code changes (prompt + merge). Requires re-generating existing L2s that have structural fields embedded. Breaking change for L2 schema.

Option 4: Pure rule-based L2 with no LLM ​

  • Pros: Zero hallucination, zero inference cost
  • Cons: Cannot generate semantic intent (purpose). The "why" requires understanding, not pattern matching. L2 exists specifically to add the semantic dimension that static analysis cannot provide.

Decision ​

We adopt Option 2 (L2 inherits L1 structure at upsert time) as the immediate fix, with a roadmap toward Option 3 (prompt rewrite + regeneration) when the pipeline next does a full re-ingest.

What changes — Phase 2 merge (immediate) ​

Between the PostgreSQL fetch (Phase 2, lines 3329) and the embed_batch call (line 3336), insert a merge step:

For each L2 row in vec_rows:
  1. Parse L2's `content` JSON → extract ONLY the `"purpose"` field
  2. Find the L1 row for the same `blob_id` (level=1, chunk_index=-1)
  3. Parse L1's `content` JSON → get the full `constructs` map
  4. Build merged JSON:
     { "purpose": l2_purpose, "constructs": l1_constructs, "language": l1_language }
  5. Replace L2's `content` with the merged JSON string

This merged content enters embed_batch and gets upserted to Qdrant's nexus_ghost collection. The L2 payload now carries the full structural fingerprint + semantic purpose in a single point.

File: main.rs, function run_enrichment(), Phase 2 (lines 3306–3402) Insert point: After line 3329 (let mut vec_rows: Vec<L1Row> = ...) and before line 3334 (let texts: Vec<String> = ...)

What changes — Prompt revision (future, next full re-ingest) ​

When we regenerate all L2 abstractions, revise the prompt template (lines 3211–3227) to ask ONLY for "purpose":

From:
  TypeScript/Angular: {"purpose": "", "classes": [], "functions": [], ...}
  
To:
  Return ONLY a JSON object with one field:
  {"purpose": "1-3 sentences describing WHAT this code does and WHY"}

The structural context from L1 is still injected for reference, but the LLM no longer generates structural fields.

What does NOT change ​

  • L1 generation pipeline (Phase 1a, ast-grep extraction) — unchanged
  • L1→nexus_ast upsert — unchanged
  • L0 chunk pipeline — unchanged
  • Phase 1b LLM prompt structure — unchanged until full re-ingest
  • Abstractions table schema — no migration needed (L2 content format changes, but it's a TEXT column)
  • Qdrant collection schemas — unchanged (nexus_ghost still has text_dense 768d)

Rationale ​

Why merge at Phase 2 instead of at generation time ​

Phase 2 is where both L1 and L2 rows are already in memory from a single SQL query. The merge is a pure data transformation with zero additional I/O — the L1 content is already loaded. Adding a separate HTTP call to fetch L1 at generation time (Phase 1b) would add latency and complexity to the critical path.

Why the prompt revision is deferred ​

The merge at Phase 2 immediately fixes the Qdrant payload quality regardless of what the LLM generates. The prompt revision is a separate optimization that reduces token costs but requires a full re-ingest to benefit existing abstractions. Deferring it avoids a breaking change mid-pipeline.

Why not a separate validation step ​

A post-hoc validation script (as originally proposed in the plan's Task 5) is useful for auditing but does not fix the root cause. The merge at Phase 2 prevents corrupted data from ever reaching Qdrant — the validation script becomes a diagnostic tool rather than a quality gate.

Consequences ​

Positive ​

  • Zero structural hallucinations in Qdrant. Every L2 point in nexus_ghost carries the deterministic ground truth from ast-grep, merged with LLM-generated purpose.
  • Self-contained Qdrant payloads. nexus_ghost points now have content containing both structural constructs and semantic purpose — no cross-collection join needed.
  • Minimal code change. Single insertion point in Phase 2, ~50 lines of Rust. No schema migrations, no collection re-creation.
  • Backward compatible. Existing L2 points in Qdrant remain valid (they just lack the merge). New points will have the enriched format.
  • Token savings (after prompt revision). Removing structural field generation from the L2 prompt reduces output tokens by ~40-60%, accelerating the pipeline by the same factor.

Negative ​

  • Merge adds CPU overhead in Phase 2 (JSON parse + rebuild). For 50-row batches, this is negligible (<1ms) compared to the LLM generation time (~200ms per chunk) and embedding time.
  • Edge case: no L1 exists. If a blob has L2 chunks but no L1 (e.g., Markdown files that ast-grep doesn't support), the merge should skip and pass L2 content through unchanged. This prevents regressions for unsupported file types.
  • Edge case: L1 changes after L2 is generated. Since L2 is generated per chunk at a point in time, and L1 is regenerated per blob on each ingest, there's a temporal skew window where L2's inherited L1 structures may be stale. This is acceptable — the same skew exists today between any L0 chunk and its parent blob.

Risks ​

  • L1 JSON format changes in a future ast-grep version could break the merge parser. Mitigation: pin ast-grep version (currently 0.43.0 in Dockerfile.coordinator) and validate the merge against known L1 JSON shapes.
  • LLM might skip the "purpose" field. If the LLM outputs malformed JSON without purpose, the merge should fall back to using the entire L2 content as-is, logging a warning. This prevents the pipeline from stalling on individual chunk failures.

Implementation Notes ​

Merge algorithm (Rust, Phase 2 insertion point) ​

rust
// After vec_rows is populated (line 3329), before embed_batch (line 3334)
// Group rows by blob_id for O(n) lookup
let l1_by_blob: HashMap&lt;uuid::Uuid, &L1Row&gt; = vec_rows
    .iter()
    .filter(|r| r.level == 1)
    .map(|r| (r.blob_id, r))
    .collect();

// Merge L2 rows with their L1 parent
for row in vec_rows.iter_mut() {
    if row.level != 2 { continue; }
    let Some(l1) = l1_by_blob.get(&row.blob_id) else { continue; };
    
    let l2_json: serde_json::Value = match serde_json::from_str(&row.content) {
        Ok(v) =&gt; v,
        Err(_) =&gt; { warn!("L2 JSON parse failed for {}", row.id); continue; }
    };
    let l1_json: serde_json::Value = match serde_json::from_str(&l1.content) {
        Ok(v) =&gt; v,
        Err(_) =&gt; { warn!("L1 JSON parse failed for {}", l1.id); continue; }
    };
    
    let mut merged = l1_json.clone();
    if let Some(purpose) = l2_json.get("purpose") {
        merged["purpose"] = purpose.clone();
    }
    row.content = merged.to_string();
}

Fallback for blobs without L1 ​

rust
// If no L1 exists for this blob_id, skip merge (use raw L2 content)
// This handles Markdown, TOML, .txt files that have L0 chunks but no L1 AST

Prompt revision (future) ​

rust

Qdrant payload (unchanged) ​

The payload at lines 3360–3368 (level, content, blob_id, abstraction_id, ingested_at, ext, chunk_index) stores row.content which is now the merged JSON. No schema change needed.
