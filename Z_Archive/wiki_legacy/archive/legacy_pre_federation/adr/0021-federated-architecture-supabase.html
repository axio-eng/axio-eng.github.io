ADR-0021: Federated Architecture (Local-First + Remote Sync) via Supabase ​

Status ​

Proposed

Context ​

Atualmente, o NEXUS opera como um ambiente estritamente local (stateful e isolado). O ecossistema é formado por nexus_engine, PostgreSQL, Qdrant, e Ollama gerenciados localmente (via Podman/Docker Compose). Embora a abordagem atual garanta latência sub-milissegundo (&lt; 1ms) e privacidade total dos dados (zero vazamento de IP da empresa para modelos fechados), ela apresenta um limitador crítico: a incapacidade de compartilhar inteligência e memória contextual entre diferentes agentes/desenvolvedores do time.

Para resolver o problema da inteligência coletiva e transformar o NEXUS em um ecossistema estilo "Agent Hive" sem perder os benefícios de latência/segurança para código não publicado, propomos uma arquitetura federada.

Decision Drivers ​

  • Inteligência Coletiva (Agent Hive): Necessidade de agentes de IA em máquinas distintas compartilharem um grafo semântico central de metadados, soluções e estrutura de código.
  • Privacidade do Código-Fonte (Local-First): Código em elaboração, chaves locais e rascunhos devem permanecer no grafo local (Local Node).
  • Consolidação Operacional: Reduzir a carga de containers locais que o desenvolvedor precisa rodar na própria máquina.
  • Complexidade do Stack Vetorial: Manter PostgreSQL e Qdrant separadamente exige maior custo de gerenciamento do que unificar vetores e relações.

Considered Options ​

Option 1: Manter estritamente Local ​

  • Pros: Zero rede; latência máxima de leitura; segurança total do IP.
  • Cons: Agentes são incapazes de aprender uns com os outros; retrabalho na descoberta de caminhos T4 para cada membro do time.

Option 2: Migrar Tudo para Cloud (Supabase Total) ​

  • Pros: Único grafo como single-source-of-truth; stack ultra-simplificado via DBaaS (Supabase pgvector + RLS); excelente interface (Studio).
  • Cons: Latência maior para buscar RAG; tráfego contínuo do código para a nuvem; inabilidade de agentes trabalharem de forma totalmente off-line/privada.

Option 3: Federação Local-First + Remote Sync (Supabase) ​

  • Pros: Retém a privacidade na ponta via grafo local (SQLite/Postgres local) para código não-commitado. Constrói um Hive remoto para a main e "Reflections" T5.
  • Cons: Maior complexidade arquitetural no MCP para rotear consultas nexus_query em paralelo; necessita de lógica de sync (Rehydration/WebSockets).

Decision ​

We will adote Option 3: Federação Local-First + Remote Sync (Supabase). O NEXUS adotará uma estratégia de dois nós:

  1. Edge/Local Node: Motor RAG rodando no host local indexando drafts e uncommitted changes.
  2. Cloud/Global Node (Supabase): Instância compartilhada contendo a "Memória Institucional" indexada via CI/CD a partir de branches principais, com arestas (Edges) que podem ser herdadas pelo grafo local.

Como primeiro passo, utilizaremos Supabase com pgvector e migraremos progressivamente as relações mantidas isoladamente entre Qdrant/Postgres para simplificar o motor no cenário remoto.

Rationale ​

Para alcançar o potencial máximo de uma Hive de Agentes, eles devem operar em uma memória distribuída. Supabase fornece as ferramentas out-of-the-box (pgvector, Realtime, Edge Functions e RLS) para transformar o banco de dados em um Message Bus semântico para Agentes. Adotando um Roteamento Híbrido, protegemos os drafts na máquina do desenvolvedor enquanto consumimos as sabedorias passadas do repositório remoto.

Consequences ​

Positive ​

  • Agentes colaborativos: Respostas passadas, links T4 e bridges semânticos criados remotamente são consultáveis por todo o time.
  • Simplicidade do Compose Remoto: Redução de infraestrutura para o nó central. Supabase unifica vetores e relacionamentos relacionais.

Negative ​

  • O roteador de buscas nexus_query precisará tratar falhas parciais (timeout no Supabase não pode quebrar a busca local).

Risks ​

  • Resolução de Conflitos T4: Um nó T4 local que foi excluído, mas o T4 remoto aponta para ele. Necessidade de Edge verification.

Implementation Notes ​

Roadmap de Execução Iterada ​

Fase 1: Preparação do Motor (Spike)

  • Implementar interface StorageAdapter no Rust Engine.
  • Isolar as chamadas SQL e chamadas ao Qdrant.

Fase 2: Conector Supabase & Roteamento Híbrido

  • Levantar nó Supabase para testes e injetar modelo "T4 Graph" usando pgvector.
  • Modificar o nexus_query no MCP para realizar Fan-out: Consulta banco local + consulta Supabase.
  • Fazer deduplicação (chunk_id) no Coordinator, prevalecendo a versão Local.

Fase 3: Cross-Graph Edges e Subscrições

  • Permitir arestas (depends_on:imports, part_of) entre o Local File Node e um Remote Chunk.
  • Implementar client de supabase-realtime no Engine para invalidar caches locais quando ocorrer um commit na main.
