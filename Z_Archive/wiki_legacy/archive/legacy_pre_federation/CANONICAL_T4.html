NEXUS Canonical T4 — Grafo Multi-Relacional de Navegação Topológica ​

Propósito ​

Este documento define o T4 em seu estado atual como grafo multi-relacional em produção — camada de navegação topológica que conecta arquivos e chunks por 4 tipos de aresta, organizados em 3 eixos de projeção. Ele substitui a versão anterior que subestimava o papel do T4 como mero estágio intermediário.

O T4 não é uma visão futura nem um experimento. É o grafo que os agentes navegam hoje via nexus_adjacent() para expandir contexto, cruzar dependências e descobrir relações estruturais e semânticas.

Para a especificação detalhada das arestas, consulte: T4_GRAPH_NORMS.md.

O que o T4 é hoje ​

O T4 é um hipergrafo multi-relacional persistido em PostgreSQL (links), alimentado por:

  • part_of — hierarquia file→chunk (extração estrutural do pipeline de ingestão)
  • depends_on:imports — dependências entre arquivos (extraídas de imports TypeScript, Rust, Python, SQL)
  • semantic_bridge — similaridade semântica entre summaries de file nodes
  • contradicts — contradições detectadas entre chunks semanticamente incompatíveis

O ponto de entrada estrutural é o file node (level=4). Todo chunk pertence a exatamente um file node via part_of. A navegação cross-file ocorre exclusivamente no nível de file nodes; chunks de arquivos diferentes não têm arestas diretas.

Os 4 tipos de aresta (criteria) ​

CriteriaDireçãoOrigemNatureza
part_ofchunk → file (inbound)Pipeline de ingestãoHierárquica — fato
depends_on:importsA → B (A importa B)Parser de imports (TS, Rust, Python, SQL)Estrutural — fato
semantic_bridgeBidirecional simétricaSimilaridade de embeddings (threshold 0.80)Semântica — aproximação
contradictsBidirecional simétricaPipeline de enriquecimentoConflito — aproximação

Cada criteria projeta um ângulo de visão diferente sobre o mesmo sistema. Um mesmo par de nós pode ter múltiplas arestas de criteria diferentes.

Os 3 eixos de projeção ​

Toda navegação no T4 é uma fatia tridimensional:

EixoValoresExemplo
criteriapart_of, depends_on:imports, semantic_bridge, contradictsSó dependências estruturais
file_typetypescript, sql, markdown, python, json, rust, yamlSó TypeScript
projectnexus, intraclinica-ngSó intraclinica

Fatia composta típica:

criteria=depends_on:imports + file_type=typescript + project=intraclinica-ng
→ "dependências entre arquivos TypeScript do frontend"

Anatomia do grafo: esqueleto, carne, cartilagem ​

O T4 tem três camadas de rigidez, como um organismo biológico:

CamadaMetáforaArestasMutabilidade
RígidaEsqueletodepends_on:importsFato — imutável. Se A importa B, a aresta existe.
SemânticaCarnesemantic_bridge, contradictsAproximação — mutável com threshold/modelo.
ClassificaçãoCartilagemfile_type, projectMetadado — classifica os nós, não as arestas.

Esqueleto: o agente confia cegamente em depends_on:imports para navegação estrutural.

Carne: semantic_bridge é útil para descoberta, mas admite falsos positivos. O agente usa como dica, não como fato.

Cartilagem: os eixos file_type e project organizam os nós em categorias. O agente fatia o grafo para focar no formato relevante.

Regra de navegação: esqueleto primeiro, carne depois — caminhe por dependências estruturais antes de explorar pontes semânticas.

O triângulo T3 → T4 → L3 ​

T3 (Qdrant)            T4 (Grafo)              L3 (Domínio)
────────────           ──────────              ───────────
busca semântica        expansão topológica     síntese emergente
encontra sementes      caminha o grafo         cluster de domínio
┌──────────┐           ┌──────────┐            ┌──────────┐
│  chunk   │───T3───▶  │ file node│───T4────▶  │ domínio  │
│ semente  │   query   │ (level4) │  caminhada  │ emergido │
└──────────┘           └──────────┘            └──────────┘
     │                      │                       ▲
     │    nexus_query()     │   nexus_adjacent()    │
     └──────────────────────┴───────────────────────┘
  1. T3 encontra sementes via busca semântica (nexus_query).
  2. T4 expande topologicamente a partir do file node (nexus_adjacent).
  3. L3 não existe mais como nível fixo no banco — emerge por detecção de comunidades no grafo T4. O agente pode agrupar file nodes conectados por depends_on:imports + semantic_bridge e tratar o cluster como um domínio coerente.

A implicação prática: L3 não é armazenado como blob individual. É uma síntese de domínio que o agente materializa on-the-fly ao caminhar o T4 e identificar subgrafos densos.

Padrões de navegação ​

Subir: chunk → file node ​

nexus_query("...") → obtém node_id do chunk
nexus_adjacent(chunk) → filtra outbound part_of → file node (level=4)

Descer: file node → chunks ​

nexus_adjacent(file_node) → inbound com criteria=part_of → lista de chunks

Cruzar por dependência: file → file ​

nexus_adjacent(file_node) → outbound com criteria=depends_on:imports
→ arquivos importados pelo file atual

Cruzar por semântica: file → file ​

nexus_adjacent(file_node) → inbound/outbound com criteria=semantic_bridge
→ arquivos com summary similar

Combinar: T3 + T4 completo ​

nexus_query("...") → chunk node_id
  → adjacent(chunk) → sobe (part_of) pro file node
  → adjacent(file) → cruza (depends_on) pra outros files
  → adjacent(file) → desce (part_of) pros chunks alvo

Direção canônica ​

O T4 é o grafo em produção. Sua evolução segue estes princípios:

  • File node (level=4) permanece como ponto de entrada estrutural. Toda navegação começa ou termina nele.
  • depends_on:imports é a espinha dorsal. Arestas estruturais são prioridade — são determinísticas, confiáveis e independem de modelo.
  • semantic_bridge é complementar. Útil para descoberta em território desconhecido; transitório por natureza.
  • L3 emerge, não é armazenado. A síntese de domínio é responsabilidade do agente via community detection, não um blob pré-computado.
  • O triângulo T3→T4→L3 é o fluxo canônico. Query semântica → expansão topológica → síntese de domínio.
