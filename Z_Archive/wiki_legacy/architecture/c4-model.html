Modelo C4 — NEXUS: Multi-Relational Graph Runtime for LLM External Memory ​

Versão: 4.0
Última atualização: 25 Jun. 2026
Propósito: Documentar a arquitetura do NEXUS nos 4 níveis do modelo C4 (Contexto, Contêiner, Componente, Código), incorporando todas as decisões arquiteturais ativas.

Level 1: Diagrama de Contexto do Sistema ​

O NEXUS é um sistema de memória externa para LLMs e agentes de IA que combina busca semântica vetorial (T3) com navegação topológica multi-relacional (T4). O sistema permite que agentes naveguem em bases de código complexas sem depender de busca por força bruta (grep/find/loops de leitura), reduzindo deriva semântica e preservando contexto operacional.

Stack Arquitetural ​

T1 Filesystem → T2 PostgreSQL → T3 Qdrant → T4 Multi-Relational Graph → T5 Reflections → T6 Time

Usuários do Sistema ​

Level 2: Diagrama de Contêineres ​

O NEXUS é composto por 5 contêineres principais. O Coordinator (Rust/tokio) é o cérebro do sistema: abriga o pipeline de enriquecimento (L0→L1→L2), o handler de consultas, as operações de administração e a lógica do grafo T4. Os Nomad Workers (ADR-0003) são contêineres lógicos dentro do Coordinator.

Nomad Trait Interface (ADR-0003) ​

rust
/// Trait fundamental para workers independentes do coordinator.
/// Cada loop de background implementa Nomad, permitindo
/// lifecycle, health e métricas individualizados.
#[async_trait]
pub(crate) trait Nomad: Send + Sync {
    /// Nome único para logging, métricas e health check.
    fn name(&self) -> &'static str;

    /// Intervalo de sleep quando não há trabalho.
    fn idle_interval(&self) -> Duration;

    /// Lógica central: observa estado e age.
    /// Retorna contagem de itens processados.
    async fn tick(&self, pool: &PgPool) -> Result<usize>;

    /// Health check opcional.
    fn is_healthy(&self) -> bool { true }
}

// Registro central — um vec em main.rs
let nomads: Vec<Box<dyn Nomad>> = vec![
    Box::new(L1Nomad::new()),
    Box::new(L2Nomad::new()),
    Box::new(ReflectionNomad::new()),
    Box::new(L3Nomad::new()),
    Box::new(WeaverNomad::new()),
    Box::new(DecayNomad::new()),
    Box::new(QdrantReconcileNomad::new()),
    Box::new(LinkValidationNomad::new()),
];

/// Entidade topológica — um subgrafo rotulado no T4.
struct Entity {
    id: Uuid,                    // Estável, imutável — o symbol_id do T4
    nodes: HashSet<NodeId>,      // Nós do grafo T4 que compõem a entidade
    edges: HashSet<Edge>,        // Arestas internas (conectividade do subgrafo)
    boundary: HashSet<Edge>,     // Arestas de fronteira (conexões externas)
    schema: TypeSchema,          // Forma abstrata que a entidade implementa
    confidence: f64,             // 1.0 = fato, 0.0 = hipótese
}

// Operadores canônicos — assinaturas
type Schema = TypeSchema;         // Forma abstrata de entidade
type Engine = String;            // "postgresql", "timescaledb", etc.
type CriteriaSet = Vec<String>;  // ["part_of", "depends_on:imports", ...]
type Axes = Map<String, String>; // {file_type: "sql", project: "nexus"}

fn find(subgraph_schema: &Schema) -> Result<Vec<Entity>>;
fn expand(entity: &Entity, criteria_set: &[&str]) -> Result<Entity>;
fn project(entity: &Entity, axes: &Axes) -> Result<Entity>;
fn abstract_entities(entities: &[Entity]) -> Result<TypeSchema>;
fn compile(form: &TypeSchema, engine: &str) -> Result<String>;
fn chain(entity: &Entity, operator: &dyn Fn(&Entity) -> Result<Entity>) -> Result<Entity>;

Decaimento Stigmergic (Forgetful Swarm) — ADR-0021 ​

rust
/// Protocolo de esquecimento estigmérgico.
/// Quando uma tríade (Agente A, Agente B, Contexto C) exibe
/// drift entrópico persistente > 0.6 por 3 ciclos consecutivos,
/// a associação é marcada como "Conflict Island".

struct ConflictIsland {
    triad: (NodeId, NodeId, NodeId),  // A, B, C
    entropy_drift: f64,               // > 0.6 por 3 ciclos
    cycles_above_threshold: u8,       // contagem de ciclos consecutivos
    decay_score: f64,                 // peso atual do decaimento
}

// Mecanismo de esquecimento: decaimento linear
fn stigmergic_decay(link: &mut AgentLink) {
    if link.decay_score > 0.0 {
        link.decay_score -= DECAY_RATE;  // DECAY_RATE ≈ 0.05 por ciclo
        if link.decay_score <= DECAY_THRESHOLD {
            // Abaixo do limiar: desaparece da visão L2/L3
            link.visible = false;
        }
    }
}

Context GC e tokens·hora (ADR-0023) ​

rust
/// Métrica de custo cognitivo: tokens processados por hora de vida.
/// Permite identificar subcontextos descartáveis que não geraram
/// decisões ou conexões úteis no grafo.
struct CognitiveCost {
    abstraction_id: Uuid,
    tokens_consumed: u64,       // Tokens de entrada/saída das chamadas Ollama
    lifetime_hours: f64,        // Tempo de vida no sistema
    links_generated: u32,       // Links no grafo que este chunk gerou
    decisions: u32,             // Decisões arquiteturais associadas
}

// Candidatos a descarte: alto tokens·hora, baixo links_generated
fn gc_candidates() -> Vec<CognitiveCost> {
    abstractions
        .iter()
        .map(|a| CognitiveCost {
            tokens_per_hour: a.tokens_consumed as f64 / a.lifetime_hours.max(1.0),
            value_ratio: a.links_generated as f64 / a.tokens_consumed.max(1) as f64,
        })
        .filter(|c| c.tokens_per_hour > GC_THRESHOLD && c.value_ratio < VALUE_MIN)
        .collect()
}

Backpropagation de Cargas (GSP) — ADR-0001 (Proposto) ​

rust
/// Algoritmo proposto baseado em Processamento de Sinais em Grafos (GSP).

// Atualização de pesos por retropropagação:
// W(e_ij)^{(t+1)} = β · W(e_ij)^{(t)} + (1-β) · ΔW_ij
// Onde β é a taxa de aprendizado (0 < β < 1) e ΔW_ij é o sinal de reforço.

// Potencial propagado com decaimento por cosseno:
// f(θ, d) = cos(θ) · e^{-α·d}
// Onde θ = similaridade de cosseno do embedding (T3),
//       d = distância topológica no grafo T4,
//       α = coeficiente de decaimento.

// Aproximação de Chebyshev (ChebNet) para filtragem espectral:
// Ordem K truncada → complexidade O(K·M), onde M = número de arestas.
// Evita fatoração espectral O(N^3).

// Operadores de wavelet espectral:
// - Translado T_v: desloca centro da consulta ao longo da AST/L1
// - Dilatação D_s: ajusta escala da busca (local → global)

Esquema do Banco (T2 — PostgreSQL) ​

sql
-- Tabela central de abstrações (NEXUS canonical store)
CREATE TABLE abstractions (
    id UUID PRIMARY KEY,
    project TEXT NOT NULL,
    level INTEGER NOT NULL,        -- 0 = L0 (raw), 1 = L1 (AST), 2 = L2 (semantic), 4 = file
    blob_path TEXT,
    chunk_index INTEGER,
    summary TEXT,
    vector_id UUID,
    metadata JSONB,
    heat REAL DEFAULT 0.0,          -- ADR-0024: propagação de calor
    last_heated_at TIMESTAMPTZ,     -- ADR-0024: timestamp do último aquecimento
    tokens_consumed BIGINT DEFAULT 0, -- ADR-0023: tokens processados
    created_at TIMESTAMPTZ DEFAULT now()
);

-- Tabela de links do grafo T4
CREATE TABLE agent_links (
    id UUID PRIMARY KEY,
    source_abs_id UUID REFERENCES abstractions(id),
    target_abs_id UUID REFERENCES abstractions(id),
    criteria TEXT NOT NULL,          -- part_of, depends_on:imports, semantic_bridge, contradicts
    weight REAL DEFAULT 1.0,
    decay_score REAL DEFAULT 0.0,    -- ADR-0021: decaimento estigmérgico
    created_at TIMESTAMPTZ DEFAULT now()
);

-- Índices
CREATE INDEX idx_abstractions_heat ON abstractions (heat) WHERE heat > 0.0;
CREATE INDEX idx_agent_links_criteria ON agent_links (criteria);
CREATE INDEX idx_abstractions_level ON abstractions (level);
CREATE INDEX idx_abstractions_project ON abstractions (project);

Nota sobre Discrepância do ADR-0001 ​

O ADR-0001 apresenta uma discrepância entre suas fontes:

  • O cabeçalho do arquivo (docs/adr/0001-backpropagation-abstractions-charges.md) indica "Proposto (Proposed)"
  • O índice geral (docs/adr/index.md) o marca como ✅ Active
  • O índice local (docs/adr/README.md) o marca como "Proposto"

Para fins deste modelo C4, o ADR-0001 é tratado como 📝 Proposto (alignado com o cabeçalho do documento), uma vez que a implementação Chebyshev/GSP/dipolo ainda não foi realizada — o que existe é o heat propagation (ADR-0024), que é uma simplificação da proposta original.

Nota sobre ADR-0025 ​

O ADR-0025 está em estágio 💭 Pensamento/Discussão. Nenhuma decisão foi tomada. As ideias incluem:

  • Múltiplas coleções Qdrant por projeção T4
  • Classificador LLM para tipo semântico (em vez de extensão de arquivo)
  • Dimensionalidade variável por tipo de dado
  • Tabelas relacionais separadas no PostgreSQL

Este ADR não deve ser tratado como aceito ou implementado.
