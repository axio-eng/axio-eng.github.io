Deep Dive: Grafo Multi-Relacional T4 ​

A camada de navegação topológica. O que separa NEXUS de um RAG qualquer.

Conceito ​

O T4 é um grafo multi-relacional onde nós são abstrações de código (chunks, ASTs, descrições semânticas, arquivos) e arestas são relações tipadas entre eles.

Diferente de um grafo de conhecimento tradicional (que tenta "entender" o domínio), o T4 modela a estrutura do código — hierarquia de arquivos, dependências de import, similaridade semântica e conflito.

Anatomia das Arestas ​

part_of ​

A espinha dorsal do grafo. Conecta chunks ao seu arquivo de origem.

Direção: Chunk → File Node (outbound do chunk, inbound do file)

Criação: Determinística — todo chunk sabe seu blob_id. Quando um blob é ingerido, o weaver cria links part_of de cada chunk para o file node L4 correspondente.

Exemplo: chunk_abc (L0, linha 10-30 de auth.rs) --part_of--> file_auth (L4, auth.rs)

depends_on:imports ​

Conecta arquivos que têm dependências de importação. Representa o esqueleto estrutural do código — a espinha dorsal que mostra "quem depende de quem".

Direção: Importador → Importado (A depends_on:imports B = A importa B)

Criação: Parser de imports específico por linguagem (Rust: use, TypeScript: import, Python: import).

Exemplo: auth.rs --depends_on:imports--> user_service.rs

semantic_bridge ​

Conecta arquivos semanticamente similares. Representa a carne do grafo — conexões que emergem de similaridade de propósito, não de estrutura.

Criação: Quando duas abstrações no mesmo projeto têm embeddings próximos em nexus_ghost (L2), um link semantic_bridge é criado entre seus file nodes.

Exemplo: patient_registration.rs --semantic_bridge--> appointment_scheduler.rs (ambos lidam com fluxo de pacientes)

contradicts ​

Conecta abstrações que expressam conflito semântico — chunks que descrevem a mesma coisa de formas diferentes ou incompatíveis.

Criação: Quando a distância vetorial entre duas abstrações do mesmo blob ou módulo excede um threshold de drift.

Exemplo: Versão antiga vs nova de uma mesma função que mudou de comportamento.

Eixos de Projeção ​

Toda navegação no T4 é uma fatia definida por 3 eixos:

EixoValoresUso
criteriapart_of, depends_on:imports, semantic_bridge, contradictsFiltrar por tipo de relação
file_typerust, typescript, sql, python, markdown...Filtrar por linguagem
projectnexus, intraclinica-ngFiltrar por projeto

Exemplo de projeção: "Me mostre só as dependências de import (criteria=depends_on:imports) em Rust (file_type=rust) do projeto NEXUS (project=nexus)".

Padrões de Navegação ​

Padrão Básico: Subir-Cruzar-Hidratar ​

python
# 1. Encontrar semente
seed = nexus_query("função de autenticação", top_k=1)
chunk_id = seed.packs[0].abstraction_id

# 2. Subir para file node
file_node = nexus_adjacent(chunk_id, criteria="part_of")[0]

# 3. Cruzar para dependências
deps = nexus_adjacent(file_node, criteria="depends_on:imports")

# 4. Cruzar para pontes semânticas
bridges = nexus_adjacent(file_node, criteria="semantic_bridge")

# 5. Coletar chunks de cada arquivo
for dep in deps:
    chunks = nexus_adjacent(dep, criteria="part_of")
    # cada chunk tem L0, L1, L2 disponíveis

Query Path (T-RAG) ​

O T-RAG (Topological RAG) é o coração da query:

  1. Embed: Query → vetor 768d via nomic-embed-text
  2. T3 Seeds: k-NN em nexus_ghost → top N abstractions L2
  3. T4 PPR Walk: A partir das seeds, caminha no grafo via arestas semantic_bridge e successor, propagando scores como PageRank personalizado
  4. T2 Hydrate: Para cada nó no subgrafo expandido, busca conteúdo completo, caminho, versão
  5. Context Pack: Materializa grupos coerentes de abstrações (central + chunks vizinhos)
