ADR-0010: ADR-010: Hierarquia de Levels de Abstração e o Papel do Tempo ​

Status ​

Accepted

Context / Detailed Analysis ​

Contexto: Discussão sobre qualidade do L3 atual e o que os levels acima de L2 deveriam ser.

Contexto ​

O pipeline NEXUS opera sobre quatro layers de abstração (L0–L3). A qualidade do L3 atual demonstrou-se insuficiente: insights gerados pelo qwen2.5-coder:1.5b por chunk individual são superficiais porque o contexto de entrada (~200 chars) é pequeno demais para derivar invariantes arquiteturais reais.

Esta decisão formaliza a semântica de cada level e estabelece como o tempo se encaixa no modelo.

Hierarquia de Levels ​

Cada level responde uma pergunta que o anterior não consegue responder sozinho.

LevelPerguntaGranularidadeModelo
L0O que está escrito?chunk (~2000 chars)— (verbatim)
L1O que existe?blob (arquivo completo)ast-grep (determinístico)
L2O que faz?chunk → semânticoGemini Flash
L3O que significa?blob → propósitoMistral local
L4O que emerge?cluster de blobs → padrõesGemini Flash / Sonnet
L5O que pode estar errado?sistema → hipóteses falsificáveisSonnet
L6O que o sistema quer ser?projeto → modelo mentalOpus / thinking models

Regra fundamental: a abstração só pode ser tão boa quanto o contexto que ela recebe. L3 alimentado por um chunk produz ruído. L3 alimentado pelos L2s de um blob inteiro produz síntese útil.

O Papel do Tempo ​

O tempo no NEXUS não é uma dimensão única — são três mecanismos complementares:

1. Tempo como evolução (já existe em T2) ​

blobs.prev_blob_id + blobs.is_current + link EVOLVED_FROM no hilbert_bridge. Captura a história de como o código mudou. Permite queries do tipo "qual era o propósito deste módulo 3 meses atrás?".

2. Tempo como decay (a implementar) ​

abstractions.last_accessed_at + abstractions.access_count existem no schema mas não são usados. A ideia: abstrações não acessadas há muito tempo perdem peso nas buscas — memória que desvanece naturalmente, como em sistemas cognitivos reais. Isso evita que insights obsoletos contaminem queries sobre o estado atual.

3. Tempo como dimensão algébrica (visão futura — ver ADR-011) ​

Em T5, o tempo seria uma dimensão do espaço de conceitos — permitindo "fatiar" o sistema em qualquer instante passado e comparar topologias. Isso requer T5 implementado.

Decisão ​

  1. L3 será reformulado para operar por blob (agrupando todos os L2s do mesmo blob), não por chunk individual. Modelo: Mistral local (já disponível na GTX).

  2. L4 será experimental — síntese cross-blob sobre clusters relacionados por similaridade vetorial ou links topológicos. Modelo: Gemini Flash.

  3. L5/L6 são visão — não implementar agora, registrar a semântica.

  4. Decay será ativado como mecanismo de ponderação nas queries T3, usando last_accessed_at. Abstrações acessadas recentemente ganham boost; antigas, penalidade.

  5. Evolução temporal (prev_blob_id) permanece como está — fundação sólida.

Adendo: Pipeline de Dois Estágios para L3 (Mistral como Provocador) ​

A qualidade do L3 depende não só do contexto (L2s por blob) mas da qualidade do prompt entregue ao modelo externo (Sonnet/Gemini). Um prompt genérico produz insights genéricos.

Arquitetura proposta ​

L2s do blob
    ↓
  Mistral local  ← semente rotacionada
  "dado esses resumos, [papel] + [foco] + [tom]"
    ↓
  prompt enriquecido + hipóteses especulativas
    ↓
  Sonnet / Gemini Flash
  "aqui o contexto + as hipóteses. valide, refute, derive invariantes."
    ↓
  L3 de alta qualidade

O Mistral não precisa ter razão — precisa fazer as perguntas certas para provocar o modelo externo a raciocinar mais fundo do que faria com um prompt genérico.

Rotação de sementes ​

Cada execução do L3 sorteia uma combinação de três dimensões:

Papel do Mistral (quem ele está sendo):

  • Cético — "o que pode estar errado aqui?"
  • Arqueólogo — "o que este código revela sobre decisões passadas?"
  • Usuário externo — "o que alguém novo precisaria saber sobre este módulo?"
  • Detetive — "o que está oculto mas presente?"
  • Físico — "quais são as forças e tensões em jogo?"

Foco (o que observar):

  • Contratos externos — o que este módulo promete ao resto do sistema
  • Invariantes internos — o que nunca pode mudar sem quebrar tudo
  • Acoplamentos implícitos — o que depende de quê sem declarar
  • Evolução provável — para onde este código parece estar indo

Tom (como formular):

  • Afirmativo — "este módulo é responsável por..."
  • Interrogativo — "por que este módulo precisa de...?"
  • Hipotético — "se X mudar, o que quebra?"

Por que rotacionar ​

Um mesmo blob executado N vezes com sementes diferentes acumula L3s com perspectivas genuinamente distintas. O L4 futuro sintetiza esses L3s divergentes numa visão mais completa do que qualquer execução única produziria — análogo a como ensemble methods em ML superam modelos individuais ao explorar o espaço de hipóteses de forma diversa.

Sem rotação, o L3 converge para um padrão fixo determinado pelo prompt — viés sistemático que se propaga para L4 e L5.

Consequências ​

  • run_reflect no Rust precisa ser reescrito para agregar L2s por blob antes de gerar L3.
  • O pipeline L3 passa a ter dois estágios: Mistral local (provocador) → modelo externo (síntese).
  • A semente (papel × foco × tom) é sorteada por execução e salva em reflections.metadata para rastreabilidade.
  • O mesmo blob pode ter múltiplos L3s com perspectivas diferentes — schema deve permitir (blob_id, seed_hash) como chave, não apenas blob_id.
  • last_accessed_at deve ser atualizado toda vez que uma abstração for retornada por nexus_query.
  • L4+ ficam documentados mas não implementados até validação do L3 reformulado.

Adendo: Batch Size e Teste A/B Ollama vs Antigravity na Ingestão L2 ​

O engine Rust processa L2 em batches de 50 abstractions por ciclo (LIMIT 50). Durante reindexações completas (ex: intraclinica-ng com potencial de 50k+ chunks), isso representa um gargalo significativo na GPU local (GTX 1660 Super, 6GB VRAM).

Opções de backend para L2 ​

BackendBatch size idealLatência por chunkCustoQuando usar
qwen2.5-coder:1.5b (Ollama local)50~200mszerodesenvolvimento, testes, chunks sensíveis
Antigravity proxy (Gemini Flash)200–500~80msquota APIingestão de projetos grandes

Teste A/B proposto ​

Durante a ingestão do intraclinica-ng, comparar:

  • Ramo A: Ollama local, batch=50, sem paralelismo externo
  • Ramo B: Antigravity proxy, batch=200–500, paralelismo controlado pelo pool de contas

Métricas a comparar: tempo total de ingestão, qualidade dos L2s (amostra manual de 20 chunks), taxa de erros, consumo de quota.

Implicações de implementação ​

  • O batch size deve ser configurável via env var ENRICHMENT_BATCH_SIZE (padrão: 50)
  • O ENRICHMENT_MODEL env já diferencia o backend; batch size maior só faz sentido com Antigravity
  • O proxy Antigravity já implementa retry exponencial e pool de contas — adequado para batches maiores
  • Resultado do teste A/B deve atualizar este ADR com o batch size canônico por backend

Adendo: Fronteira Fractal — L3 Contextualizado por Vizinhos Topológicos ​

Ideia ​

O pipeline L3 atual recebe apenas os L2s do blob atual para gerar o insight. A hipótese fractal propõe que o modelo de L3 receba também fragmentos dos L3s dos blobs topologicamente vizinhos (via hilbert_bridge ou links de supersession), expandindo o contexto como uma fronteira fractal — não linearmente, mas explorando as arestas do grafo de conhecimento já existente.

Propriedade autorreferente ​

O modelo usa conhecimento interno (L2s + L3s já gerados) para gerar hipóteses sobre um novo blob, e essas hipóteses alimentam futuros L3s que se tornam contexto para próximas execuções. A fronteira cresce usando apenas o que o sistema já sabe — sem dados externos.

É análogo a como um fractal gera complexidade a partir de uma regra simples aplicada recursivamente sobre seu próprio output: L3(blob_n) = f(L2s(blob_n) + L3s(vizinhos(blob_n))).

O que isso resolve ​

O L3 atual é "cego" ao grafo — gera insights isolados por blob. Com vizinhança topológica, o L3 de um guard Angular passaria a receber o L3 do serviço que ele protege e do módulo que o declara, produzindo insights que atravessam camadas arquiteturais.

Estado atual ​

Não implementado. O hilbert_bridge não está populado no pipeline v5 atual. A implementação requer:

  1. Popular hilbert_bridge com arestas de co-ocorrência/import entre blobs
  2. No ciclo L3, fazer JOIN hilbert_bridge para recuperar L3s dos N vizinhos mais próximos
  3. Injetar fragmentos dos L3s vizinhos no prompt do modelo de reflexão antes do L2 atual

Próximos passos ​

  • Avaliar se hilbert_bridge deve ser populado pelo engine Rust (watcher de imports) ou offline (análise de grafo)
  • Definir N (número de vizinhos) e estratégia de truncamento para caber na janela de contexto
  • Registrar resultado em ADR dedicado quando implementado
