ADR-0002: Operadores Relacionais Composáveis no T4 para Compilação Topológica de Esquemas ​

Status ​

Implementado parcialmente — Operadores REST expostos via engine.

  • nexus_find → Não implementado (rota 404)
  • nexus_expand → Não implementado (rota 404)
  • nexus_project → POST /t4/project (projeção do grafo multi-relacional) ✅
  • nexus_abstract → POST /t4/abstract (fingerprinting Weisfeiler-Lehman) ✅
  • nexus_compile → POST /t4/compile (compilação de fingerprints em schemas) ✅
  • nexus_chain → POST /t4/chain (pipeline runtime, campo esperado: steps não operations) ✅

Nota: find e expand nunca foram implementados como endpoints. A navegação é feita via project (subgrafo por critério) + expand não existe — usa-se project com diferentes parâmetros.

Contexto ​

O NEXUS opera com dois sistemas de referência cruzada:

  1. T3 (Qdrant/Espaço Vetorial): Busca semântica aproximada por similaridade de cosseno. Responde "o que parece com isto?" — útil para descoberta, mas sem garantia de correção estrutural.
  2. T4 (Grafo Multi-Relacional/PostgreSQL): Arestas exatas e aproximadas entre nós de abstração (part_of, depends_on:imports, semantic_bridge, contradicts). Três camadas de rigidez: esqueleto (fatos estruturais), carne (aproximações semânticas), cartilagem (classificação por eixos criteria × file_type × project).

O pipeline atual de enriquecimento (L0→L1→L2→Qdrant upsert + weaver) extrai relações estruturais no nível de chunks de código — funções, classes, imports. O T4 hoje navega entre file nodes e chunks, mas as entidades do grafo são fragmentos de arquivos, não estruturas relacionais completas.

Paralelamente, a ferramenta Codanna demonstrou um padrão eficaz de operadores relacionais composáveis:

find_symbol("Parser")
  → get_calls(parser)       # o que Parser chama
  → find_callers(parser)    # quem chama Parser
  → analyze_impact(parser)  # grafo completo de dependências
  → chain(symbol_id)        # profundidade via composição

Cada operador é uma transformação pura: recebe um conjunto de entradas (símbolos), retorna um conjunto de saídas (relacionados). A composição de operadores forma uma lei (query) que pode ser reaplicada em diferentes contextos. A chave do Codanna não é o que cada operador faz individualmente — é que os operadores são composáveis, tipados (cada operador tem entrada/saída bem definida), e encadeáveis (o output de um alimenta o input do próximo via symbol_id).

O desafio que se coloca agora é: como elevar este padrão de operadores composáveis ao nível topológico (T4), onde as entidades não são mais símbolos de código, mas estruturas relacionais inteiras?

O Problema Central ​

Em bases de código tradicionais, as relações são explícitas: imports, chamadas de função, herança de classe. Em datasets massivos de alto valor (múltiplos schemas SQL, bases documentais heterogêneas, corpora multi-lingua), as entidades e relações não são dadas a priori — elas precisam ser descobertas, abstraídas e formalizadas.

  • Um schema SQL de 200 tabelas com milhares de colunas, FKs, índices, views materializadas, triggers, RLS policies — a "entidade" relevante pode ser um agregado de 12 tabelas interligadas por FKs e regras de negócio, não uma tabela individual.
  • Um dataset de documentos jurídicos com citações cruzadas, hierarquias normativas, revogações tácitas — a "relação" relevante pode ser uma cadeia de precedência que cruza 30 documentos, não uma referência direta.
  • Um corpus de telemetria de microserviços com 500 endpoints, 2000 chamadas RPC, latências e dependências não-lineares — a "estrutura" relevante pode ser um subgrafo de serviços que compartilham estado, não um grafo de chamadas plano.

Precisamos de operadores que trabalhem no nível topológico — que recebam uma estrutura relacional inteira como entrada e produzam uma nova estrutura relacional como saída — com garantias consistentes de endereçamento e composição.

Decision Drivers ​

  • Composabilidade: Operadores devem ser encadeáveis como funções puras — a saída de um operador deve poder alimentar o próximo sem adaptação manual.
  • Consistência de Endereçamento: Cada entidade no T4 precisa de um identificador estável e imutável (UUID) que persista através de transformações — análogo ao symbol_id do Codanna, mas para estruturas relacionais inteiras.
  • Invariância de Nível: Operadores devem preservar ou explicitar o nível de abstração. Um operador que cruza tabelas de um schema SQL opera no nível schema; um que cruza schemas entre bancos opera no nível federation. A composição entre níveis diferentes deve ser mediada por operadores de projeção/reificação explícitos.
  • Independência de Domínio: Os operadores devem funcionar para schemas SQL, documentos jurídicos, grafos de telemetria — qualquer domínio onde existam entidades relacionais identificáveis.
  • Tratamento de Incerteza: Diferente do depends_on:imports (fato estrutural), operadores de descoberta operam com thresholds de similaridade. O sistema deve expor a confiança de cada aresta produzida, permitindo que o agente decida se confia ou não.

Considered Options ​

Opção 1: Operadores Específicos por Domínio (Status Quo Estendido) ​

Criar operadores especializados para cada tipo de dado: get_tables(schema), get_foreign_keys(table), get_callers(endpoint). Cada operador com semântica própria, sem interface unificada.

  • Pros: Simples de implementar para cada domínio isoladamente. Baixo custo inicial.
  • Cons: Não composável. Cada dupla de operadores precisa de adaptação manual. A "lei" (query composta) não é reutilizável entre domínios. Não resolve o problema de abstração topológica — apenas empurra a complexidade para o agente.

Opção 2: Operadores Genéricos com Projeção por Eixos (Extensão do T4 Atual) ​

Generalizar o padrão atual do T4 — onde criteria × file_type × project define fatias do grafo — para criar operadores genéricos que operam sobre qualquer combinação de eixos:

find_entities(criteria=part_of, project=intraclinica-ng, file_type=sql)
  → traverse(target="outbound", criteria="depends_on:imports")
  → project(axis="file_type", value="typescript")
  → impact_analysis(depth=3)
  • Pros: Reutiliza a estrutura de eixos já existente no T4. Operadores independentes de domínio. Composição natural (saída = conjunto de nós no grafo, que alimenta o próximo operador).
  • Cons: O modelo atual do T4 opera no nível de file nodes e chunks, não no nível de estruturas relacionais completas. Um conjunto de tabelas SQL que formam um agregado de domínio não é um nó — são múltiplos nós conectados por arestas. Operadores genéricos precisariam de clusterização para tratar múltiplos nós como uma entidade única.

Opção 3: Operadores Topológicos com Entidades como Subgrafos (Proposta) ​

Elevar o nível de abstração: em vez de operar sobre nós individuais, os operadores operam sobre subgrafos rotulados — conjuntos de nós + arestas que formam uma estrutura relacional coerente.

# Operadores fundamentais no nível topológico

find_entity(subgraph_schema)        → entidade: subgrafos que casam o schema
  → expand(entity, criteria_set)    # expande o subgrafo por tipos de aresta
  → project(entity, axes)           # projeta em sub-eixos (fatia)
  → abstract(entities)              # generaliza: encontra forma canônica entre subgrafos
  → compile(abstract_form, target)  # materializa: gera schema concreto alvo
  → chain(id)                       # conecta saída de um operador na entrada do próximo

Onde cada entidade topológica é definida por:

Entity {
  id: UUID,               // estável, imutável — o symbol_id do T4
  nodes: Set<NodeID>,     // nós do grafo T4 que compõem a entidade
  edges: Set<Edge>,       // arestas internas (conectividade do subgrafo)
  boundary: Set<Edge>,    // arestas de fronteira (conexões com o exterior)
  schema: TypeSchema,     // forma abstrata que a entidade implementa
  confidence: f64         // grau de certeza (1.0 = fato, 0.0 = hipótese)
}

Pros:

  • Composabilidade real: operadores são transformações Entity → Entity
  • Endereçamento consistente: UUID estável persiste através da cadeia
  • Independência de domínio: tabelas SQL, documentos jurídicos, microserviços — tudo são subgrafos
  • A lei (composição de operadores) pode ser serializada, versionada, reaplicada
  • A operação compile(abstract_form, target) permite gerar schemas concretos a partir de formas abstratas descobertas em datasets massivos

Cons:

  • Maior complexidade de implementação: requer detecção de subgrafos, operações de junção/projeção em grafos
  • Clusterização de nós em entidades requer algoritmo de community detection ou seed-based expansion
  • A noção de "schema" de uma entidade topológica precisa ser formalizada (TypeSchema)

Opção 4: Operadores Híbridos T3+T4 (Exploração Vetorial + Expansão Topológica) ​

Em vez de operar puramente no grafo T4, usar T3 para encontrar sementes semanticamente similares e T4 para expandir estruturalmente — o triângulo canônico T3→T4→L3, formalizado como operadores composáveis.

semantic_seed(query, top_k=5)        # T3: busca vetorial → N sementes
  → structural_expand(seed, depth=3) # T4: caminha grafo a partir de cada semente
  → cluster(subgraphs)               # L3: detecta comunidades → entidades emergentes
  → abstract(entities)               # generaliza entidades → forma canônica
  → compile(form, engine="sql")      # materializa forma em schema SQL
  • Pros: Aproveita o melhor dos dois sistemas — busca semântica para descoberta (T3) + expansão exata para estrutura (T4). O L3 emerge naturalmente como community detection.
  • Cons: Herda a incerteza do T3 (threshold de similaridade) que contamina a cadeia. A composição entre operadores T3 e T4 requer conversão explícita de coordenadas (espaço vetorial ↔ grafo).

Decisão ​

Adotaremos a Opção 3 (Operadores Topológicos com Entidades como Subgrafos) como direção arquitetural, com os seguintes operadores fundamentais:

Operadores Canônicos ​

Seja ε o conjunto de todas as entidades topológicas (subgrafos rotulados) no T4:

OperadorAssinaturaDescrição
find(subgraph_schema)Schema → P(ε)Encontra subgrafos no T4 que casam um schema de entidade
expand(entity, criteria_set)ε × P(C) → εExpande a fronteira do subgrafo por tipos de aresta
project(entity, axes)ε × P(A) → εFatia o subgrafo por eixos de projeção
abstract(entities)P(ε) → FGeneraliza múltiplas entidades em uma forma canônica
compile(form, engine)F × Engine → SchemaMaterializa uma forma canônica em schema concreto
chain(entity, operator)ε × O → εConecta saída de um operador na entrada do próximo

Relação com os Eixos T4 Existentes ​

Os operadores não substituem os eixos atuais do T4 (criteria × file_type × project) — eles os encapsulam. O operador project(entity, axes) é uma generalização do conceito de fatia: em vez de projetar sobre file nodes, projeta sobre qualquer entidade topológica, usando os mesmos eixos como filtros.

O operador expand(entity, criteria_set) corresponde a caminhar o grafo T4 por tipos específicos de aresta — exatamente como nexus_adjacent(file_node) funciona hoje, mas com a diferença fundamental de que a expansão pode parar na fronteira da entidade, distinguindo arestas internas (conectividade do subgrafo) de arestas de fronteira (conexões com o exterior).

A Operação abstract: O Coração da Compilação Topológica ​

A operação abstract(entities) recebe um conjunto de entidades (subgrafos) e produz uma Forma Canônica — uma generalização que captura o padrão invariante entre elas. Este é o mecanismo que permite:

  1. Descobrir formas abstratas em datasets massivos: Dado um dataset de 500 schemas SQL de diferentes sistemas, abstract(schemas) produz uma forma canônica que captura o padrão relacional comum — "todo sistema de agendamento tem uma tabela appointments com FKs para patients e providers, um status state machine, e um índice temporal para evitar conflitos de horário."

  2. Compilar formas abstratas em schemas concretos: Dado a forma canônica do parágrafo anterior, compile(form, engine="postgresql") gera um schema SQL inicial com as tabelas, colunas, FKs, índices e constraints que implementam o padrão — que pode então ser refinado manualmente.

  3. Generalizar entre domínios: Uma forma canônica descoberta em schemas SQL pode ser comparada com formas descobertas em schemas MongoDB (collections, embedded documents, indexes) — revelando isomorfismos estruturais entre paradigmas de banco diferentes.

O algoritmo de abstract proposto:

  1. Alinhar os subgrafos por isomorfismo aproximado (kernel de Weisfeiler-Lehman)
  2. Identificar nós e arestas consistentes entre todos os subgrafos (invariantes)
  3. Parametrizar nós/arestas variantes (ex: nomes de tabela específicos → slots nomeados)
  4. Produzir a forma canônica como um grafo com nós parametrizados e arestas invariantes

Endereçamento e Composição: A "Lei" como Query ​

Inspirado pelo symbol_id do Codanna, cada entidade topológica carrega um UUID estável. A composição de operadores forma uma lei — uma query topológica serializável:

// Exemplo: "Encontre schemas de agendamento que usam PostgreSQL
//          e compile um schema inicial para TimescaleDB"

find(schema="appointment_system_v1")
  → expand(criteria=["depends_on:imports", "semantic_bridge"])
  → project(axes={file_type: "sql"})
  → abstract()
  → compile(engine="timescaledb")

Esta lei pode ser versionada, compartilhada entre agentes, e reaplicada em datasets diferentes. A diferença fundamental para uma query SQL tradicional é que a lei opera sobre topologia relacional, não sobre tuplas.

Rationale ​

Por que a Opção 3 e não a 1 ou 2? ​

  • Opção 1 (operadores específicos por domínio) falha na composabilidade — exatamente o problema que o Codanna resolveu com symbol_id universal. Se cada operador tem sua própria semântica de entrada/saída, não há "lei" reutilizável.

  • Opção 2 (operadores genéricos com projeção por eixos) é um passo intermediário correto, mas insuficiente. O T4 atual opera sobre nós individuais (file nodes, chunks), não sobre estruturas relacionais completas. A noção de "entidade como subgrafo" é a diferença entre ter um operador find_tables(schema) que retorna tabelas individuais e ter find_aggregate(schema) que retorna o subgrafo completo de tabelas interligadas que formam um agregado de domínio.

Por que não a Opção 4? ​

A Opção 4 (híbrido T3+T4) é uma extensão futura válida, mas introduz dependência de similaridade vetorial no pipeline de compilação. A incerteza do T3 contamina a cadeia de operadores, exigindo propagação de confiança. A Opção 3 é mais pura: opera exclusivamente no grafo topológico, onde as arestas são fatos (esqueleto) ou aproximações explícitas (carne). O T3 entra como seed opcional na ponta do find, não como parte da cadeia de operadores.

Por que UUID como identificador universal? ​

O Codanna usa symbol_id numérico. No T4, UUID é necessário porque:

  1. Entidades topológicas podem ser compostas por múltiplos nós — UUID desacopla identidade de localização
  2. UUID permite fusão: se dois subgrafos são reconhecidos como a mesma entidade em momentos diferentes, o UUID permanece estável
  3. UUID é imutável através de transformações — expand(entity) preserva o UUID da entidade original

Consequências ​

Positivas ​

  • Composabilidade real: Operadores Entity → Entity podem ser encadeados arbitrariamente. A "lei" (composição) é serializável como JSON e reaplicável.
  • Independência de domínio: O mesmo conjunto de operadores funciona para schemas SQL, documentos jurídicos, grafos de telemetria. A diferença está no conteúdo do subgrafo, não na forma do operador.
  • Descoberta em datasets massivos: abstract(entities) permite extrair formas canônicas de datasets de milhares de schemas — um mecanismo de engenharia reversa de bancos de dados em escala.
  • Compilação de schemas: compile(form, engine) gera schemas concretos a partir de formas abstratas, acelerando a criação de novos bancos a partir de padrões descobertos na natureza.
  • Integração natural com o T4 existente: Operadores reutilizam os eixos criteria × file_type × project como filtros de projeção. A tabela agent_links (93.957 arestas) já contém a matéria-prima para expand e find.

Negativas ​

  • Complexidade de abstract: O algoritmo de generalização de subgrafos (kernel de Weisfeiler-Lehman) é computacionalmente caro em datasets grandes. Exige implementação cuidadosa em Rust.
  • Definição de TypeSchema: A noção de "schema de entidade topológica" precisa ser formalizada antes da implementação. Uma entidade SQL é diferente de uma entidade documental — o TypeSchema precisa ser um tipo algébrico que capture ambas.
  • Consumer da lei: A "lei" (composição de operadores) precisa de um runtime de execução — um interpretador de pipelines topológicos. Isso é um componente novo na arquitetura.

Riscos ​

  • Super-engenharia antecipada: O formalismo de operadores pode ser excessivo se o número de domínios for pequeno. Deve ser validado com pelo menos 3 domínios distintos antes de generalizar.
  • Performance de abstract: Weisfeiler-Lehman em subgrafos grandes pode ser O(n²) no pior caso. Mitigação: usar amostragem (top-k subgrafos por seed) e paralelização.
  • Acoplamento T4: A implementação depende da maturidade do T4. Se o T4 ainda está em transição (ex: semantic_bridge sendo substituído), os operadores precisam ser resilientes a mudanças de schema de arestas.

Implementation Notes ​

  1. Fase 1 — Operadores de Navegação (find, expand, project): Implementar primeiro os operadores que são generalizações diretas do T4 existente. Reutilizar nexus_adjacent como backend. Adicionar o conceito de entidade como coleção de nós (subgrafo). Protótipo em Python puro para validação.

  2. Fase 2 — Operador abstract: Implementar kernel de Weisfeiler-Lehman para alinhamento de subgrafos. Protótipo em Python com NetworkX. Validar com 3 datasets: schemas SQL de intraclinica-ng, schemas NEXUS, schemas públicos de código aberto.

  3. Fase 3 — Operador compile: Mapear formas canônicas → DDL SQL (PostgreSQL, TimescaleDB). Seguir o padrão já estabelecido em intraclinica-ng (migrations numeradas, CREATE TABLE ..., FKs, índices, RLS). A saída deve ser um migration SQL válido.

  4. Fase 4 — Chain runtime: Implementar o interpretador de pipelines topológicos em Rust, exposto como MCP tools (nexus_compose, nexus_discover, nexus_compile).

A.3 Lições dos Artigos para o T4 ​

  1. Parcimônia primeiro: O teorema da aproximação universal (Funayama, 1997) sugere que uma única camada intermediária bem projetada é suficiente. No T4, isso significa que expand não deve ser usado indiscriminadamente — a arquitetura mínima que resolve o problema é a arquitetura correta.

  2. Múltiplos algoritmos, múltiplos resultados: Castro et al. (1999) mostraram que A*, CASCOR e PPL produzem arquiteturas diferentes para o mesmo problema. O T4 deve expor múltiplos operadores de descoberta (equivalentes a diferentes estratégias construtivas) para que o agente escolha a mais adequada ao domínio.

  3. Critério de parada ≈ Threshold de cristalização: O MSE=0.01 define quando o A* para de crescer. No T4, o equivalente é o limiar de confiança da Fractal Boundary — que determina quando uma entidade transita de Fluida para Cristalizada. Sem critério de parada, o grafo cresce indefinidamente.

  4. "Construtivo" não significa "sempre crescer": A* e CASCOR frequentemente não adicionaram neurônios — a arquitetura inicial já era suficiente. O T4 deve suportar a decisão de não expandir.

  5. PPL como inspiração para project: O PPL projeta dados em subespaço revelando estrutura latente. É análogo direto ao operador project(entity, axes) — mudar o eixo de observação para revelar relações que não são visíveis no eixo original.
