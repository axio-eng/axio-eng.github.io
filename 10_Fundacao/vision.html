Documento de Visão — NEXUS ​

Versão: 1.0 — Junho de 2026
Status: Rascunho

1. Introdução ​

1.1 Propósito ​

Este documento define a visão de alto nível do NEXUS — Multi-Relational Graph Runtime for LLM External Memory. Ele descreve o problema que o sistema resolve, os stakeholders envolvidos, as capacidades essenciais e a proposta de valor do produto.

1.2 Escopo ​

O NEXUS é um sistema de memória externa para agentes de IA (LLMs) que combina busca semântica vetorial com navegação topológica em grafo multi-relacional. O sistema permite que agentes de IA naveguem em bases de código complexas sem depender de busca por força bruta (grep/find/loops de leitura), reduzindo deriva semântica e preservando contexto operacional.

2. Contexto de Negócio ​

2.1 O Problema ​

Agentes de IA modernos (LLMs) enfrentam limitações críticas ao navegar em bases de código complexas:

  1. Janela de contexto limitada — impossível carregar uma base de código inteira.
  2. Deriva semântica — o agente alucina estruturas que não existem.
  3. Custo de busca — loops de grep/find consomem tokens e janela de contexto.
  4. Perda de memória operacional — soluções passadas são esquecidas entre sessões.
  5. Falta de navegação topológica — busca semântica plana não entende hierarquia de código.

2.2 A Solução ​

O NEXUS resolve esses problemas através de uma arquitetura bipartida:

  • T3 (GPS Semântico) — embeddings vetoriais em espaço 768d que entendem intenção e proximidade semântica. O agente pergunta "onde fica a lógica de agendamento?" e o GPS aponta a vizinhança.
  • T4 (Mapa da Cidade) — grafo multi-relacional hierárquico com arestas de hierarquia (part_of), dependência (depends_on:imports), similaridade (semantic_bridge) e conflito (contradicts). O agente navega pela estrutura real do código.

3. Visão Geral do Produto ​

3.1 Arquitetura em Uma Linha ​

T1 Filesystem → T2 PostgreSQL → T3 Qdrant → T4 Multi-Relational Graph → T5 Reflections → T6 Time

3.2 Capacidades Essenciais ​

  1. Ingestão Multi-Repo: Monitora mudanças no filesystem, ingere novos arquivos e atualiza automaticamente o grafo.
  2. Pipeline de Abstração em 3 Níveis:
    • L0: Chunking determinístico por limites naturais (AST-aware).
    • L1: Extração estrutural via ast-grep (funções, classes, interfaces).
    • L2: Geração de descrições semânticas via LLM local (Ollama).
  3. Navegação Topológica (T4): Caminhada multi-relacional com 4 tipos de aresta e 3 eixos de projeção.
  4. Busca Híbrida (T3 + T4): Embedding → seeds vetoriais → expansão no grafo → hidratação de contexto.
  5. Memória Stigmergic: Agentes registram descobertas via nexus_link(), criando trilhas para agentes futuros.
  6. Reflexões Arquiteturais (L3): Síntese de domínio emergente por detecção de comunidades no grafo.
  7. MCP Server: Interface padronizada SSE para qualquer cliente compatível.

4. Premissas e Dependências ​

4.1 Premissas ​

  1. Ollama como motor de inferência local — modelo qwen2.5-coder:nexus para geração L2/L3, nomic-embed-text para embeddings.
  2. GPU NVIDIA disponível (mín. 6GB VRAM) — GTX 1660S, 29 layers offloaded.
  3. PostgreSQL 16 com pgvector — schema evolutivo via migrations.
  4. Qdrant como store vetorial — coleções nomeadas com vetor text_dense (768d, Cosine).
  5. Rust como linguagem de implementação — performance e segurança de memória.
  6. ast-grep como extrator estrutural — análise sintática determinística.
